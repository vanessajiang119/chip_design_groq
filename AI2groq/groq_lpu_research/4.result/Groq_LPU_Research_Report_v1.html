1. TSP 架构深度分析 Tensor Streaming Processor

Groq 的 Tensor Streaming Processor（TSP）采用了一种与传统 GPU/CPU 截然不同的架构范式——确定性计算（Deterministic Computing）。其核心理念是在硬件层面完全消除所有反应式元件（reactive elements），包括 cache 层次结构、分支预测器、乱序执行引擎、动态仲裁器以及投机执行单元等传统处理器中不可或缺的组件。这些功能全部被前移到编译阶段，由编译器在离线程完成静态调度。这种设计选择使得每一指令的执行时间在编译时即可精确预知到时钟周期级别，彻底消除了传统架构中因动态硬件调度带来的延迟抖动（jitter）问题，这是 Groq LPU 能够实现极低推理延迟的根本原因。

TSP 最具创新性的设计是其垂直功能切片（Functional Slicing）组织方式。芯片被划分为五种不同类型的垂直列（slice）：ICU（Instruction Control Unit，指令控制单元）、MEM（Memory Slice，内存切片）、VXM（Vector Execution Module，向量执行模块）、MXM（Matrix Execution Module，矩阵执行模块）和 SXM（Shift Execution Module，移位执行模块）。数据流沿 East-West 方向在相邻功能切片之间直接传递，而指令流沿 North-South 方向从顶部的 ICU 向下分发到各 slice。每个 slice 类型在芯片上重复排列 20 次（tile），每个 tile 包含 16 条 SIMD 通道，总计 320 条并行通道。数据直接在 producer 和 consumer 之间流动，完全不需要经过寄存器文件或 cache 层次，从而实现了极低的延迟（<49μs 单图推理）和极高的带宽利用率（~80 TB/s SRAM 带宽）。

确定性设计原则

Eliminating all reactive hardware
  • 去除 cache 层次 — SRAM 直接寻址，编译器静态分配地址空间
  • 去除乱序执行 — 144 个独立指令队列由编译器全权控制发射时序
  • 去除分支预测 — 采用 predicated execution，无分支预测失败惩罚
  • 去除动态仲裁 — 所有路由路径和时序在编译时预计算完成
  • 结果：接近 0% 的功耗浪费在硬件调度逻辑上

Streaming 数据流模型

Producer-consumer dataflow
  • 320 SIMD lane × 64 逻辑 stream（32 Eastbound + 32 Westbound）
  • Producer-Consumer 模式 — 数据从 MEM slice 产生后直接流向 compute slice
  • 无寄存器文件 — 数据在 stream 上流动时即刻被消费
  • Compiler 可 cycle-accurate 追踪每个 stream 在芯片上的精确位置
  • 结果：ResNet-50 单 batch 20,400 img/s，延迟 <49μs

2. 核心论文与开源资源 Papers & Open Source

Groq 的技术路线通过两篇 ISCA 论文公开发表：2020 年的 TSP 基础架构论文（"Think Fast: A Tensor Streaming Processor for Accelerating Deep Learning Workloads"）首次阐述了功能切片和确定性调度的核心理念；2022 年的多处理器扩展论文（"A Software-Defined Tensor Streaming Multiprocessor for Large-Scale Machine Learning"）进一步描述了如何通过 Dragonfly 拓扑和软件定义网络将数千个 TSP 组织为单一逻辑计算核心，该论文获得了 ISCA 2022 的会议奖项。在软件层面，Groq 开源了 GroqFlow 编译工具流（基于 MLIR 前端 + Haskell 后端的双层架构），支持将 PyTorch/TensorFlow 模型自动编译为 GroqChip 上可执行的指令序列。

特别需要指出的是，Groq 并未开放任何硬件 RTL 代码（Verilog/VHDL/Chisel），其芯片设计全部为 proprietary 性质。对于希望学习或复现类似架构的研究者而言，最可行的路径是基于开源 systolic array 项目（如 svkocherla/systolic_array 或 vbachh43/Transformer_Accelerator_Based_on_FPGA）结合 ISCA 论文中描述的调度原理，构建一个功能简化但架构理念一致的参考实现。社区项目 LPU-sims 提供了一个基于 Python 的 LPU 推断估算原型，可以作为理解 TSP 调度行为的起点。

3. 编译器设计分析 Compiler Stack

Groq 的编译器是其技术栈中最为关键、技术含量最高的部分，承担了传统 CPU/GPU 中由硬件完成的所有动态调度工作。编译器采用 MLIR 前端 + Haskell 后端 的两层架构：前端负责将 PyTorch/TensorFlow 等框架的模型计算图转换为 MLIR（Multi-Level Intermediate Representation）中间表示，并在此阶段进行算子融合、内存分配等高级优化；后端则使用 Haskell 语言编写的领域特定语言（DSL）进行 cycle-accurate 的指令调度和资源分配。后端需要同时管理 144 个独立指令队列的发射时序、320 条 SIMD lane 上 64 个逻辑 stream 的 East-West 路由路径，以及片间 Dragonfly 拓扑中数千个 TSP 之间的数据传输调度。

在数值精度管理方面，Groq 提出了 TruePoint Numerics 技术：编译器根据每个算子的精度敏感性自动选择最优的数值格式——Attention logits 使用 FP32 保证精度，MoE 权重使用 Block Floating Point 平衡效率与精度，激活层使用 FP8 最大化吞吐。所有中间累加结果使用 100-bit 定点格式保证无损累加，最终输出再截断到目标精度。编译器还集成了形式化验证（temporal logic + model checking）来确保调度结果正确性。

4. 三天执行计划 Execution Plan

以下计划基于"理解 Groq 设计理念 + 复用开源 systolic array + 聚焦 Transformer 算子"的核心原则制定。整体思路是从开源 Verilog systolic array 项目出发（其功能对应 TSP 中的 MXM matrix slice），逐步构建一个功能简化但架构理念一致的 mini Groq-like 原型系统。三天分别聚焦硬件基础（Day 1）、编译器原型（Day 2）和系统集成与验证（Day 3），每天产出可独立验证的交付物，最终目标是在模拟器上跑通一个简化版 Transformer 的 Attention 算子。

鉴于 Groq 的许多技术细节（如 Haskell DSL 的具体实现、144 指令队列的精确仲裁逻辑、TruePoint 100-bit 累加器硬件实现等）属于 proprietary 信息，本计划中的实现方案将采用"功能等价替代"原则——即用开源社区已有的成熟组件（如 standard systolic array + PyTorch + Verilator 仿真）构建一个可运行的参考系统，重点验证确定性调度和 streaming dataflow 这两个核心理念。

D1

架构学习 + RTL 设计

8-10 h · 交付物: rtl/pe.v, rtl/systolic_array.v, rtl/tb_systolic_array.v

上午精读 ISCA 2020 和 2022 两篇核心论文，建立对 TSP 功能切片架构和确定性调度原理的完整理解。下午基于 svkocherla/systolic_array 等开源项目实现参数化 PE（Processing Element）模块和可配置 N×N systolic array（N=4/8/16），采用 weight-stationary 数据流。使用 Verilator 完成仿真验证，并集成 AXI4-Stream 接口方便后续与 CPU 侧通信。

D2

编译器原型开发

8-10 h · 交付物: compiler/scheduler.py, compiler/dsl.py, compiler/test_matmul.py

上午安装 GroqFlow（pip install groqflow）并理解其 MLIR → Haskell 的编译流程和数据表示方式。下午在 Python 中实现一个轻量级静态指令调度器（cycle-accurate 资源分配，跟踪每个数据 stream 在 systolic array 中的位置），并设计一个简易 DSL 用于描述 load→compute→store 三阶段 pipeline。将编译生成的指令序列通过 Verilator 加载到 Day 1 的 RTL 模型上运行并验证正确性。

D3

系统集成 + Transformer 算子验证

8-10 h · 交付物: transformer/attention.sched, demo/run_demo.sh

上午完成 RTL + 编译器 + Python 软件栈的系统级联调，确保端到端链路通畅。下午将 Attention 算子的核心计算步骤（Q×K^T MatMul → Softmax → Score×V MatMul）逐一映射到 systolic array 上，实现完整的计算图调度。在一个小型 Transformer 模型（约 4K 参数，1 层，8 个 head，64 dim）上跑通完整推理流程，对比仿真延迟与 Groq 官方公布的性能基线，分析系统瓶颈。
