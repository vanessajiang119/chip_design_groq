## 1. 确定性调度的核心原理

### 1.1 静态调度 vs 动态调度

Groq TSP 的核心区别在于将所有调度决策从硬件运行时移至编译器编译时：

```
传统 CPU/GPU:                    Groq TSP:
┌────────────────────┐          ┌────────────────────┐
│ 编译器: 生成代码    │          │ 编译器: 生成代码 +  │
│                    │          │  静态调度所有指令   │
└────────┬───────────┘          └────────┬───────────┘
         ▼                               ▼
┌────────────────────┐          ┌────────────────────┐
│ 硬件运行时:          │          │ 硬件: 严格按计划执行 │
│  OoO 调度器          │          │  无任何运行时决策    │
│  分支预测器          │          │  无调度器            │
│  Cache 层次         │          │  无仲裁器            │
│  动态网络路由        │          │  无 cache           │
└────────────────────┘          └────────────────────┘
```

### 1.2 Groq 编译器如何实现静态调度

编译器在编译时即完全确定：

**a) 指令时序 (Instruction Timing)**
- 每条指令的延迟在硬件中是固定的、已知的
- 编译器知道精确到 cycle 的指令完成时间
- 无需像 OoO 那样依赖 ROB (Reorder Buffer) 跟踪指令状态

**b) 数据到达时间 (Data Arrival Time)**
- 每个张量从 MEM slice 读取到送达 VXM/MXM/SXM 的时间精确已知
- 编译器确保数据和指令在同一 cycle 到达同一功能单元

**c) 网络路由 (Network Routing)**
- 所有数据包的发送时间和路径在编译时确定
- 无需 request/response 协议
- 发送方和接收方的时序精确对齐

## 2. 编译器调度算法

### 2.1 144 指令队列的调度

TSP 有 144 个独立指令队列，编译器需要管理：

| 维度 | 数量 | 说明 |
|------|------|------|
| 指令队列数 | 144 | 每个队列独立发射 |
| SIMD 通道 | 320 | 20 tiles × 16 lanes |
| 逻辑流 | 64/通道 | 32 East + 32 West |
| 片上 SRAM | 220 MB | 全局共享 |
| 主频 | 900 MHz | 每个 cycle 都被精确调度 |

### 2.2 ILP (指令级并行) 调度

TSP 被描述为一个 **"144-wide VLIW 架构"** — 编译器在 144 个指令槽中打包指令并调度：

- 每 cycle 每个功能切片可以发射多条指令
- 编译器负责指令打包，无硬件重排序
- 指令和数据的"相遇"发生在确定的 cycle 和位置

### 2.3 关键调度约束

```
功能切片每 cycle 指令分配:
  MXM (矩阵):  6 条指令
  SXM (开关):  14 条指令
  MEM (内存):  44 条指令
  VXM (向量):  16 条指令
  ─────────────────────────
  合计:        80+ 条指令/cycle 可并行发射
```

编译器必须确保：
1. 每个功能切片在其指令带宽限制内被调度
2. 数据依赖关系在所有队列之间正确同步
3. 流缓冲区不出现上溢/下溢
4. 所有队列在 barrier/sync 点上对齐

## 3. 同步机制: SYNC / NOTIFY

### 3.1 基本同步指令

- **SYNC**: 等待所有先前的指令完成 (全局屏障)
- **NOTIFY**: 通知其他队列某个事件已完成
- 编译器通过精确 timing 最小化同步开销

### 3.2 硬件对齐计数器 (HAC)

多芯片扩展时的全局同步机制：
- 每个 TSP 有一个 256-cycle 计数器
- 芯片间通过链接同步 HAC 值，建立**全局时间域**
- **SAC (软件对齐计数器)**: 追踪本地时间
- **DESKEW / RUNTIME_DESKEW**: 重新对齐所有 TSP

## 4. "复杂性守恒" 定律

一个关键洞察：**复杂性不会消失，只是转移**：

```
GPU:     硬件运行时状态丰富 (S_hw ↑)  →  编译器工作量小 (S_compiler ↓)
Groq TSP:硬件运行时状态极小 (S_hw ↓)  →  编译器工作量大 (S_compiler ↑)
```

### 4.1 GPU 上的复杂性 (动态调度)
- **Warp Scheduler**: 硬件逻辑选择每个 cycle 发射哪个 warp
- **Scoreboarding**: 追踪跨 warp 的寄存器依赖
- **ROB (Reorder Buffer)**: 追踪 OoO 完成和 in-order 退休
- **Cache Coherence**: 窥探、MESI 协议、总线仲裁
- **分支预测**: 推测执行 + 误预测恢复
- **动态资源划分**: SM 划分、线程块调度

### 4.2 Groq TSP 上的复杂性 (静态调度)
- **编译器需要完整的延迟模型**: 所有指令、内存、网络的精确时序
- **全局调度问题**: 144 队列的协同调度是 NP-hard 的
- **静态内存分配**: 所有 SRAM 地址在编译时确定
- **流水线平衡**: 所有功能单元的利用率优化
- **流管理**: 缓冲区大小和流 ID 的全局分配

## 5. 优势与代价

### 5.1 确定性执行节省了什么？

| 节省项 | GPU 上的开销 | TSP 上的状态 |
|--------|------------|-------------|
| **控制逻辑面积** | ~20-30% die area | <3% (ICU) |
| **调度能耗** | 每 cycle warp 调度逻辑 | 0 (编译时已完成) |
| **Cache MISS 能耗** | 动态，取决于访问模式 | 无 cache |
| **推测执行能耗** | 分支预测 + 误预测恢复 | 无推测执行 |
| **仲裁能耗** | 总线、网络、内存仲裁 | 无仲裁器 |
| **数据搬运能耗** | 数据在 cache 层次间移动 | 显式流式搬运 |

**关键数据**: 确定性执行和数据流控制可节省 60-80% 的通常浪费在数据搬运和控制开销上的能量（来源: ISCA 2020）。

### 5.2 确定性执行的代价

| 代价项 | 具体影响 |
|--------|---------|
| **编译器复杂度** | GroqFlow 编译器非常复杂，使用 MLIR + Haskell 后端 |
| **动态工作负载** | 不规则/稀疏计算因无法静态预测而性能下降 |
| **编程约束** | 需要显式流管理，开发者需理解数据流模型 |
| **灵活性损失** | 无法从硬件动态调度中受益（如分支丰富的代码） |
| **大模型容量** | 220 MB SRAM 远小于 GPU HBM (80 GB+)，需要模型并行 |
