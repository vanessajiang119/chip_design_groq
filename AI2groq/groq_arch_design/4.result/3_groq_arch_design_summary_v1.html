设计哲学 / Design Philosophy

Groq TSP 的设计围绕三个核心原则展开，这些原则构成了与传统 CPU/GPU 设计范式的根本性决裂。

1. 确定性执行 (Deterministic Execution)

完全消除所有响应式硬件组件：无 OoO、无分支预测、无缓存、无仲裁器、无 DRAM 控制器。编译器在编译时完成所有调度决策，硬件只按预定时间表执行。

控制逻辑面积
<3%
传统 CPU 控制面积
15-25%

2. 复杂性守恒 (Complexity Conservation)

复杂度不会消失，只会从硬件转移到编译器。TSP 将调度、仲裁、路由的复杂度从硬件转移到 MLIR + Haskell 编译器，实现硬件极大简化。

3. 软件定义硬件 (Software-Defined Hardware)

编译器拥有硬件资源的完全可见性：144队列、64流寄存器、220MB SRAM 地址、所有延迟 → 编译时决定一切，硬件零运行时决策。

数据流模型: 正交流 (Orthogonal Dataflow)

指令流: 南北方向 (North from ICU), 每个 cycle 传播一个 Super Lane

数据流: 东西方向 (East/West), 通过功能切片水平流动 (32 bytes/lane/cycle)

计算触发: 指令和数据在精确预定的 cycle 在功能切片处交汇

链式执行 (Chaining): 中间结果不写入 SRAM, 直接在功能单元间流动, 节省 ~10 cycles 和 ~10 pJ 每链式步骤

Wavefront 传播: 20 个 SL 的流水线呈"波纹"式执行, 稳态下所有 SL 同时处于流水线不同阶段

编译器消除死锁

Groq 从设计层面消除死锁：无自适应路由（全静态路径）、无 request/response 协议（预协调 send/receive）、确定性定时（精确知道 buffer 写入/读取时间）。无需传统 VC shifting（节省 20-30% buffer 面积）。

3天实现开发计划 / 3-Day Implementation Plan

基于上述架构分析，制定以下 3 天开发计划，目标为：完成 Groq TSP 的文档设计 / RTL 设计 / 软件开发 / 编译器设计，并跑通一个小的 Transformer 算子。

Day 1 — 环境搭建与架构基线

文档设计 + RTL 顶层骨架 + 开发环境

任务交付物

1.2 根据架构分析报告，编写 TSP 顶层模块接口定义（SystemVerilog）tsp_top.sv — 含 ICU/MXM/VXM/MEM/SXM 五个子模块接口
1.3 设计指令集架构（ISA）草案 — 6 类指令格式（Load/Compute/Activate/Shift/Sync/NOP）tsp_isa.html
1.4 搭建 GroqFlow 风格 MLIR 编译器前端框架 — 定义 TSP Dialect 基本 opstsp-dialect/ 基础结构
1.5 编写 1-SL（Super Lane）的数据流仿真 Python 模型sim/single_sl.py
Day 2 — 核心模块实现

RTL 模块设计 + 编译器调度原型 + 算子映射方案

任务交付物

全链集成 + Transformer 算子 bring-up + 验证
