## 一、编译器调度框架

### 1.1 Groq 编译器层次

```
ML Model (PyTorch/TensorFlow)
    ↓
┌─────────────────────┐
│ GroqFlow Frontend   │  ← MLIR dialect 转换
│  (MLIR-based)       │
└─────────┬───────────┘
          ↓
┌─────────────────────┐
│ IR Optimization     │  ← 计算图优化、张量分解
│  (Haskell DSL)      │
└─────────┬───────────┘
          ↓
┌─────────────────────┐
│ Static Scheduler    │  ← 网络调度核心
│  (Haskell)          │
└─────────┬───────────┘
          ↓
┌─────────────────────┐
│ Code Generator      │  ← 生成 TSP 指令
│  (Haskell → binary) │
└─────────┬───────────┘
          ↓
TSP 指令流（确定性 cycle 精确）
```

### 1.2 网络调度的输入与输出

**输入**：
- ML 模型的计算图（DAG of operations）
- TSP 数量与网络拓扑描述
- 每 TSP 的 SRAM 分配方案
- 延迟约束（如流水线平衡要求）

**输出**：
- 每个 TSP 的完整指令流
- 每条 C2C 链路在每 cycle 的精确传输计划
- 所有 send/receive 指令的时间戳

## 二、静态路由分配算法

### 2.1 路由问题形式化

编译器面临的是 **全局通信调度优化问题**：

```
给定:
  - G(V, E): Dragonfly 拓扑图（V = TSPs, E = C2C 链路）
  - C: 通信需求集合 { (src, dst, size, deadline) }
  - BW(e): 每条链路 e 的带宽

目标:
  为每个通信需求 c ∈ C 分配:
    - 路径 P(c) ⊆ E（经过的链路集合）
    - 时间槽 T(c) ⊆ ℕ（传输的 cycle 区间）

约束:
  - 链路容量: ∀e ∈ E, Σ(T(c) on e) ≤ BW(e) 
  - 无死锁: 所有路径的依赖关系无环
  - 延迟满足: deadline(c) ≥ end_time(c)

目标函数:
  最小化最大完成时间（makespan）或最大化吞吐
```

### 2.2 算法策略（推测）

基于 Groq 的设计文档和相关研究，推测其算法策略：

```
Phase 1: 拓扑感知的任务映射
  ├── 分析 ML 模型的通信模式
  ├── 将计算任务映射到物理 TSP
  └── 利用 Dragonfly 的 group 局部性（将通信密集的任务放在同 node）

Phase 2: 最短路径分配
  ├── 为每个 (src, dst) 对选择最小跳数路径
  ├── Dragonfly: 最小 3 跳（2 local + 1 global）
  └── 路径多样性: 同一 group 内可选择不同出口 TSP

Phase 3: 时间槽分配
  ├── 将所有通信需求按时间线排列
  ├── 检测链路资源冲突
  └── 通过插入延迟或切换路径解决冲突

Phase 4: 全局验证
  ├── 验证所有 buffer 无溢出
  ├── 验证所有时序约束满足
  └── 验证无条件（路径/时间上）无冲突
```

### 2.3 路径多样性利用

Groq 编译器关键的差异优势：

```
通信需求: TSP_A → TSP_C（跨机架）

Option 1（直达）:   A───global───C           [3 hops, 低延迟]
Option 2（间接）:   A→B→global→D→C          [4 hops, 中等延迟]
Option 3(A→X)    :   A→B→X→global→Y→D→C     [5 hops, 高延迟]

传统动态网络:
  只能选择 Option 1（自适应路由不确定）
  如果 Option 1 拥塞，性能急剧下降

Groq 编译器:
  可同时使用 Option 1 + Option 2（如果带宽足够）
  编译时精确消除冲突
  可以同时使用全部路径利用所有链路
```

## 三、确定性定时模型

### 3.1 Cycle 精确执行

Groq TSP 的每条指令的执行时间在编译时完全确定：

```
指令示例:
  LOAD R1, [addr]     // 精确已知: 3 cycles
  MAC MXM, R1, R2     // 精确已知: 4 cycles  
  STORE [addr], R3    // 精确已知: 2 cycles
  C2C_SEND link=3, stream=5  // 精确已知: 21 cycles（320B 传输）
```

### 3.2 网络传输的定时

C2C 链路的传输时间由以下确定：

```
传输延迟公式:
  T_send = Data_size / (Lane_speed × Lanes × Encoding_efficiency)

  Groq TSP:
  T_send = 320 bytes / (25 Gbps × 4 × ~0.97)
         ≈ 320B / 12.125 GB/s
         ≈ 26.4 ns
         ≈ 24 cycles (@ 900 MHz)
```

### 3.3 全局时间视图

编译器维护整个系统的 **统一时间视图**：

```
Global Time (cycles)
  0    1000   2000   3000   4000   5000
  ─┼─────┼──────┼──────┼──────┼──────┼──→
TSP_A: [compute] [send] [compute] [recv]
TSP_B: [compute] [recv ] [compute] [send]
TSP_C: [compute] [recv ] [compute] [relay]
TSP_D: [compute] [send] [compute] [compute]

编译器精确到每个 cycle 对齐所有 TSP 的指令流。
漂移补偿由 HAC 在运行时处理，对编译器透明。
```

## 四、链路负载均衡的定量分析

### 4.1 负载均衡目标

在最大系统（10,440 TSPs）中，编译器需要管理的资源：

```
资源规模:
  链路总数: 10,440 × 11 / 2 ≈ 57,420 条 C2C 链路
  虚拟通道: 每条链路 ~15 GB/s 带宽
  并发流数: 每 TSP 64 个 logical streams
  调度窗口: 通常数百至数千 cycles

这是一个大规模组合优化问题，
等效于带时间约束的边着色问题（NP-hard）。
```

### 4.2 近似最优调度（推测）

Groq 编译器可能使用 **启发式 + 局部搜索** 的组合策略：

```
1. 初始调度: 贪婪算法（按通信量大小排序分配）
2. 冲突解决: 对于检测到的链路冲突：
   a. 尝试偏移时间槽（增加延迟）
   b. 尝试切换路径（绕行）
   c. 尝试拆分为更小的传输单元
3. 迭代优化: 对关键路径进行局部重排
```

由于编译器不需要实时响应，可以花费 **任意时间** 来优化调度。

### 4.3 链路利用率

Groq 公布的带宽利用率数据：

| 配置 | 链路利用率 | 注释 |
|------|-----------|------|
| 小消息（几十 KB） | 75-90% | 远优于传统网络（30-50%） |
| 中消息（几百 KB） | 80-90% | 持续保持高效率 |
| 大消息（> 1 MB） | 85-90% | 接近理论峰值 |

## 五、编译器与硬件协同设计

### 5.1 编译器感知的硬件设计

Groq 的硬件设计大量简化，因为编译器承担了传统由硬件完成的工作：

| 传统硬件功能 | Groq 实现 | 硬件节省 |
|-------------|----------|---------|
| 路由表 | 编译器静态路由 | 无硬件路由表 |
| 包分类器 | 编译器分配 stream ID | 简化了输入处理 |
| 流控逻辑 | 编译器保证时序 | 无硬件流控 |
| 重排序缓冲 | 编译器保证顺序传输 | 无重排序缓冲 |
| 死锁检测/恢复 | 编译时验证 | 无死锁检测硬件 |

### 5.2 硬件提供的原语

为了支持编译器调度，硬件提供了精确可控的原语：

```
C2C_SEND link, stream, size  // 精确控制发送时间
C2C_RECV link, stream        // 精确期望接收
DESKEW                       // 多芯片同步点
HAC_READ/WRITE               // 访问硬件对齐计数器
SYNC/NOTIFY                  // TSP 内部队列同步
```

## 六、网络调度对 ML Workload 的影响

### 6.1 LLM 推理的通信模式

Transformer-based LLM 的典型通信模式：

```
Attention Layer:
  ┌─ Q projection ─→ AllReduce ─→ ...
  └─ K projection ─→ AllReduce ─→ ...
  └─ V projection ─→ AllReduce ─→ ...

关键: 小消息（几百 KB）+ 密集 AllReduce
      编译器可精确调度这些操作，
      避免 AllReduce 期间的链路冲突。
```

### 6.2 流水线并行中的通信

对于跨多个 TSP 的流水线并行：

```
Stage 1 → Stage 2 → Stage 3 → Stage 4
  (TSP1-2)  (TSP3-4)  (TSP5-6)  (TSP7-8)

编译器可安排:
  - 每 stage 的计算时间完全相等（流水线平衡）
  - stage 间数据传输在计算间隙完成
  - 零额外通信延迟（overlap 完全）
```

### 6.3 与 GPU 集群通信对比

| 方面 | GPU 集群（NVIDIA） | Groq 系统 |
|------|-------------------|----------|
| 通信库 | NCCL（运行时优化） | 编译器静态调度 |
| AllReduce 实现 | Ring/Tree（运行时选择） | 编译器选择最优拓扑 |
| 延迟 | 受拥塞影响 | 确定性 |
| 可预测性 | 低（需要 profiling） | 高（编译时已知） |
| 小消息效率 | 低 | 高 |

## 七、关键结论

1. **编译器是 Groq 系统的智能核心** — 传统网络硬件负责的路由/流控/死锁避免，全部由编译器在编译时完成
2. **网络调度 = NP-hard 优化问题** — 但编译器可以花任意时间求解，无需实时响应
3. **路径多样性利用是独家优势** — 同时使用多条路径而不冲突，传统网络无法做到
4. **小消息性能突出** — 消除握手 RTT 对 LLM 推理的常见小张量传输极为有利
5. **硬件极度简化** — 编译器承担路由/流控/死锁避免，硬件只需提供精确可控的原语
