二、链路层协议

2.1 无数据包头的传输

与传统高速互联不同，Groq 的 C2C 链路 不使用传统数据包头：

传统数据包格式（如以太网/IP）:
┌─────────┬──────────┬──────────┬──────────┬─────────┐
│ MAC Hdr │ IP Hdr   │ TCP Hdr  │ Payload  │ CRC     │
│ 14-54 B │ 20-60 B  │ 20-60 B  │ <=1500B  │ 4B      │
└─────────┴──────────┴──────────┴──────────┴─────────┘
    总开销: 58-178 字节（对于小消息 > 20%）

Groq C2C 传输格式:
┌──────────────────────────────────────────────────┐
│ Ctrl(少量) │ Payload(320B向量)                 │
│            ┌────────────────────────────────────│
│            │ 320 × FP8/INT8/FP16 elements       │
│            └────────────────────────────────────│
└──────────────────────────────────────────────────┘
    开销: ~2.5%

2.2 控制位段（推测）

控制段可能包含：

  • Stream ID — 标识数据所属的流（0-63）
  • Sequence number — 用于重组和顺序保证
  • ECC/CRC — 错误检测/纠正
  • Type marker — 数据/同步/控制帧标识

    • 2.3 2.5% 控制开销的定量分析

    假设: 控制段 ≈ 8-10 字节，数据段 = 320 字节
开销率 = 8 / (320 + 8) ≈ 2.44%

对比:
  - 传统以太网: 20-50%（小消息）  
  - InfiniBand: 5-10%
  - NVLink: 3-5%
  - Groq C2C: ~2.5%（业界最低之一）

    三、同步协议

    3.1 全局时钟同步：HAC 机制

Hardware Alignment Counter（HAC）是 Groq 多芯片同步的核心：

HAC 同步流程:
┌──────┐                ┌──────┐
│Parent│                │Child │
│ TSP  │                │ TSP  │
└──┬───┘                └──┬───┘
   │                        │
   │── HAC_Value ─────────→│
   │                        │── 记录接收时间 ─┐
   │←── Echoed_HAC ───────│  计算往返延迟    │
   │                        │←──────────────┘
   │── 重复 N 次 ─────────→│
   │    (取平均校准)        │
   │                        │
   │── 周期 HAC 更新 ─────→│── 调整本地 HAC ─┐
   │                        │  抵消时钟漂移    │
   │                        │←──────────────┘

3.2 DESKEW 指令协议

DESKEW 指令的多 TSP 行为：

所有 TSP 同时执行:
  1. 执行 DESKEW
  2. 暂停所有后续指令
  3. 等待本地 HAC 溢出
  4. 所有 TSP 在预设的全局时间点同步恢复

RUNTIME_DESKEW（运行时变体）:
  1. 暂停到下一个 HAC 同步周期
  2. 重新校准本地时间与全局时间的偏差
  3. 补偿累积的时钟漂移
  4. 恢复执行

四、数据面协议

4.1 流传输控制

Groq 的流式传输使用 显式的 send/receive 指令：

编译时生成的指令序列（伪代码）:

// TSP 1 (Producer)
[cycle 1000] SRAM_READ addr=0x1000 → stream_buf
[cycle 1010] C2C_SEND stream_id=5, link=3
             // 在 cycle 1010 开始发送 320B

// TSP 2 (Consumer)
[cycle 1200] C2C_RECV stream_id=5, link=3
             // 在 cycle 1200 开始接收 320B
[cycle 1210] stream_buf → VXM unit

时序精度：编译器保证 TSP 2 在 cycle 1200 时，TSP 1 发送的数据已到达。这个 timing 考虑了：

  • 链路传播延迟
  • 中间转发 TSP 的处理延迟
  • 可能的时钟漂移补偿

    • 4.2 中间节点转发

当数据需要经过中间 TSP 转发时：

TSP_A → (link 3) → TSP_B → (link 7) → TSP_C

TSP_B 的指令序列（编译时生成）:
[cycle X]   C2C_RECV from TSP_A, link=3, stream=5
[cycle X+δ] C2C_SEND to TSP_C, link=7, stream=5
            // δ 极小：仅需 1-2 个 cycle 的转发延迟

因为 TSP_B 不需要检查包头或做路由决策（编译时已决定），转发几乎是 直通（cut-through） 的。

4.3 多播/广播支持（推测）

基于队列广播机制（参考单 TSP 的 NOTIFY 指令），多 TSP 通信可能支持：

  • 单播（Unicast） — 点对点传输
  • 多播（Multicast） — 一对多（如 AllReduce 的 Reduce-Scatter 阶段）
  • 广播（Broadcast） — 一对所有

编译时决定哪个 TSP 实例化哪个通信模式。

五、延迟定量分析

5.1 延迟组成

端到端延迟 = 传输延迟 + 链路传播延迟 + 中间转发延迟

TSP 间通信延迟分解:
┌─────────────────────────────────────────┐
│ 传输延迟 (320B at 15 GB/s) = ~21 ns    │
│ 链路传播延迟 (0.75m copper) = ~3 ns    │
│ 链路传播延迟 (10m AOC) = ~50 ns        │
│ 中间转发延迟 (1-3 hops) = ~5-15 ns     │
│ 时钟同步余量 = ~10 cycles = ~11 ns     │
├─────────────────────────────────────────┤
│ 合计: 节点内 = ~40 ns                   │
│ 合计: 跨机架（直接） = ~120 ns          │
│ 合计: 最大（5跳） = < 500 ns           │
└─────────────────────────────────────────┘

5.3 小消息延迟优势

对于 LLM 推理常见的 小张量（KB ~ MB 级别）：

消息大小   | Groq 延迟   | NVLink 延迟（估计） | 优势
10 KB      | ~0.7 μs     | ~3-5 μs            | 4-7x
100 KB     | ~7 μs       | ~10-15 μs          | 1.5-2x

Groq 在小消息上的突出优势来自 无需握手和路由建立。

六、Memory Semantics（内存语义）

6.1 全局内存模型

Groq 的全局内存是 逻辑上共享、物理上分布的内存模型：

  • 每个 TSP 贡献 220-230 MiB 片上 SRAM
  • 全局 SRAM 池的访问延迟 < 3 μs（任意 TSP）
  • 无 cache coherency（不需要 — 编译器保证一致性）
  • 访问模式由编译器静态决定

    • 6.2 远程内存访问

远程 SRAM 读写通过 C2C 链路 + 中间 TSP 转发：

TSP_A 读 TSP_C 的 SRAM 地址 0x2000:

L1: TSP_A 发送 read request（含地址 + stream ID）
L2: 经 TSP_B 转发至 TSP_C
L3: TSP_C 的 MEM slice 读取 SRAM
L4: TSP_C 发送数据返回
L5: 经 TSP_B 转发回 TSP_A

全部由编译器调度，无运行时请求/响应协议

6.3 Bulk Synchronous 模型

Groq 系统遵循 Bulk Synchronous Parallel (BSP) 风格的执行模型：

Phase 1: Compute — 所有 TSP 本地计算
Phase 2: Communicate — 数据在 TSP 间传输
Phase 3: Barrier — 利用 DESKEW 完成同步

编译器精确划分这些阶段，
在 Phase 2 中调度所有通信指令。

七、关键结论

  1. 极简消息格式 — 2.5% 控制开销是业界最低之一
  2. 无握手机制 — 编译器预协调 send/receive 消除往返延迟
  3. Plesiosynchronous 同步 — 独立时钟源 + HAC 漂移补偿 = 全局锁步假象
  4. Cut-through 转发 — 中间 TSP 无需路由决策，延迟极小
  5. 小消息优势显著 — 在 LLM 推理的典型小张量场景中比传统互联快 4-7x
