1. Module Overview

\n

本章节提供 SPI2AXI Bridge 模块的概要描述，包括模块标识、功能目标、非功能目标、顶层端口分组汇总以及功能边界的界定。SPI2AXI Bridge 作为 SoC 内部 AXI 总线与外部 SPI 主机之间的桥接 IP，承担着协议转换、跨时钟域数据传递和系统配置访问等核心职责。

1.2 一句话摘要

\n

SPI2AXI Bridge 是一个将 SPI 从设备接口 (SPI Slave) 转换为 AXI4-Lite 主设备接口 (AXI4-Lite Master) 的桥接 IP 核，允许外部 SPI 主机通过低引脚数串行接口访问 SoC 内部的 AXI 总线映射存储器和外设配置空间。该桥接器通过双时钟域 FIFO 实现可靠的跨时钟域数据传输（Clock Domain Crossing, CDC），并支持标准 SPI 与 Quad SPI (QSPI) 两种工作模式，为芯片调试、固件更新和系统配置提供灵活的高效通道。

\n\n

1.3 功能目标

\n

SPI2AXI Bridge 的核心功能目标包括以下五个方面：

\n
    \n
  • 支持标准 SPI (1-wire) 和 Quad SPI / QSPI (4-wire) 两种工作模式，通过控制状态寄存器（Control and Status Register, CSR）可配置切换，满足不同带宽需求的场景
    • \n
  • 将 SPI 接收到的操作码 (8-bit) 和地址 (32-bit) 解码并转换为 AXI4-Lite 读写事务，实现串行协议到并行总线协议的无缝转换
    • \n
  • 通过 dual-clock FIFO 实现 SPI 时钟域到 AXI 时钟域的可靠跨时钟域数据传输 (CDC)，确保异步时钟域间的数据完整性和亚稳态防护
    • \n
  • 支持可配置的地址 Wrap 功能，在 AXI4-Lite 单次传输约束下模拟循环地址访问，特别适用于 FIFO 轮询和连续地址扫描场景
    • \n
  • 提供寄存器文件 (CSR) 用于工作模式配置、状态查询和参数配置，所有寄存器均通过 SPI 操作码直接编址访问
    • \n
\n\n

1.6 功能边界

\n

功能边界明确划定了 SPI2AXI Bridge 的设计范围。范围内涵盖 SPI 命令解析（操作码+地址解码）、SPI 到 AXI 的读写事务转换、跨时钟域数据传输 (CDC FIFO)、地址 Wrap 循环访问以及 SPI 工作模式的动态配置。范围外包括 AXI4-Full burst 支持（本设计仅采用 AXI4-Lite single transfer）、SPI 主设备模式、SPI Dual-wire 模式以及协议级重传或 ECC 保护等高级特性。

2.2 SPI 从设备接口

\n

SPI 从设备接口是 SPI2AXI Bridge 的串行数据入口，支持标准 SPI (1-wire) 和 Quad SPI (4-wire) 两种工作模式。接口同步于外部 SPI 主机提供的 spi_sclk 时钟，最高工作频率为 50 MHz。协议采用 SPI 模式 0 (CPOL=0, CPHA=0)，即串行时钟 (sclk) 空闲状态为低电平，数据在 sclk 上升沿采样、下降沿输出。数据传输顺序为 MSB first（最高有效位在前）。

\n\n

SPI 帧格式根据访问类型分为四种：内存写帧格式为 [Opcode (8-bit)] + [Address (32-bit)] + [Write Data (32-bit)]；内存读帧格式为 [Opcode (8-bit)] + [Address (32-bit)] + [Dummy Cycles (可编程)] + [Read Data (32-bit)]；寄存器写帧格式为 [Opcode (8-bit)] + [Write Data (32-bit)]；寄存器读帧格式为 [Opcode (8-bit)] + [Dummy Cycles] + [Read Data (32-bit)]。其中，Dummy Cycles 的数量可通过 CSR 中的 DUMMY_CFG 寄存器进行编程配置，实际等待周期数为配置值加 1。

\n\n \n \n\n

2.3 AXI4-Lite 主设备接口

\n

AXI4-Lite 主设备接口是 SPI2AXI Bridge 连接 SoC 内部 AXI 总线矩阵的并行协议接口。本设计采用 AXI4-Lite 协议，它是 AXI4 协议的轻量级子集，不支持 burst 传输，固定突发长度 (AxLEN) 为 0（即 single transfer），突发大小 AxSIZE 固定为 2（32-bit / 4-byte 传输）。数据位宽固定为 32-bit，与 AXI4-Lite 协议规范一致。

\n\n

AXI4-Lite 接口包含五个独立的通道：写地址通道 (AW)、写数据通道 (W)、写响应通道 (B)、读地址通道 (AR) 和读数据通道 (R)。每个通道采用独立的 valid-ready 握手协议，通道间无顺序依赖。本设计使用 3-bit ID 宽度 (AWID/ARID/WID/RID/BID)，以满足 SoC 级 ID 追踪需求。AXI4-Lite 不支持 out-of-order 事务完成，所有事务按发起顺序完成。

\n\n

2.4 中断接口

\n

中断接口输出一个单比特电平中断信号 intr_o，该信号同步于 AXI 时钟域 (axi_aclk)。中断事件源主要包括 AXI 事务错误（如 SLVERR 或 DECERR 响应）等异常状况。中断清除机制采用读状态寄存器清除或写 1 清除（Write-1-to-Clear, W1C）方式，具体实现将在微架构设计 (LLD) 中详细定义。

\n\n

3. 顶层架构框图

\n

本章节从系统层面展示 SPI2AXI Bridge 的内部模块划分和互联关系。架构框图清晰地呈现了两个时钟域（SPI 时钟域和 AXI 时钟域）的划分、六个核心子模块的职责边界以及模块间的关键数据通道。整体架构遵循 CDC FIFO 隔离设计原则，将 SPI 协议处理逻辑和 AXI 协议处理逻辑分别置于各自的时钟域中，仅在必要时通过 dual-clock FIFO 进行数据交换。

\n\n \n\n

3.1 模块内部结构

\n\n

SPI2AXI Bridge 的内部架构可划分为 SPI 时钟域和 AXI 时钟域两大区域，通过三路独立的 dual-clock FIFO（命令 FIFO、写数据 FIFO、读数据 FIFO）实现跨时钟域隔离和数据传输。

\n\n

SPI 时钟域 (spi_sclk) 包含以下逻辑单元：\n spi_slave_if（SPI 串行接口收发与命令解析）、fsm_ctrl（主状态机，控制 SPI 到 AXI 的整个事务流程）以及 csr_regfile（配置/状态寄存器文件）。SPI 时钟域的所有时序逻辑均同步于外部 SPI 主机提供的 spi_sclk。

\n\n

AXI 时钟域 (axi_aclk) 包含以下逻辑单元：\n axi_master_if（AXI4-Lite 主接口协议处理，负责五个通道的握手协议）和 wrap_addr_gen（Wrap 地址生成和管理模块）。AXI 时钟域的所有时序逻辑均同步于 SoC 提供的 axi_aclk。

\n\n

顶层还包含 Clock / Reset / CDC Synchronizer 模块，负责两个时钟域的复位同步和全局时钟管理。

4. 数据流与控制流

\n

本章节详细描述 SPI2AXI Bridge 在读写操作中的完整数据流路径和控制流状态迁移。数据流部分按写操作和读操作分别阐述从 SPI 引脚到 AXI 总线的串行→并行→协议封装的完整链条。控制流部分通过主状态机 (FSM) 的状态迁移图展示 SPI 命令解析到 AXI 事务完成的完整流程，并分析反压传播路径和并发冲突处理策略。

\n\n

4.1 数据流路径

\n\n

写操作数据流路径如下：SPI 引脚接收串行数据 → spi_slave_if 进行串并转换（serial-to-parallel conversion）→ 命令/地址写入 cmd_fifo，写数据写入 wdata_fifo → 经过 CDC 同步到 AXI 时钟域 → axi_master_if 从 FIFO 读取命令和数据 → 发起 AXI4-Lite 写事务 (AW 通道发送地址 → W 通道发送数据 → B 通道接收响应)。

\n\n

读操作数据流路径如下：SPI 引脚接收串行数据 → spi_slave_if 串并转换 → 命令/地址写入 cmd_fifo → 经过 CDC 同步到 AXI 时钟域 → 在 SPI 域插入可编程 dummy cycles（等待 AXI 读数据返回） → axi_master_if 从 cmd_fifo 获取地址 → 发起 AXI4-Lite 读事务 (AR 通道发送地址 → R 通道接收数据) → 读数据写入 rdata_fifo → 经过 CDC 同步回 SPI 时钟域 → spi_slave_if 从 rdata_fifo 读取数据 → 并串转换 (parallel-to-serial conversion) → 通过 spi_sdo 串行发送给外部 SPI 主机。

\n\n \n \n\n

4.2 控制流 — 主状态机

\n\n

主状态机 (FSM) 位于 SPI 时钟域，控制从 SPI 命令接收到 AXI 事务完成的完整流程。FSM 共包含 8 个状态：IDLE（空闲等待）、OPCODE（接收 8-bit 操作码）、ADDR（接收 32-bit 地址）、DUMMY（读路径上的虚拟等待周期）、DATA（32-bit 数据传输）、AXI_WR（AXI 写事务执行）、AXI_RD（AXI 读事务执行）和 DONE（事务完成，返回 IDLE）。

4.3 反压与流控策略

\n

SPI2AXI Bridge 在数据流路径上实现了端到端的反压传播机制，确保在 AXI 总线拥塞或 FIFO 满溢时能够妥善处理。写操作的反压传播路径为：AXI 总线反压 → axi_master_if 等待握手 → wdata_fifo 满（无法写入新数据）→ SPI 域暂停数据发送。读操作的反压传播路径为：rdata_fifo 空 → SPI 域等待读数据 → axi_master_if 等待 R 通道响应。FIFO 的水位管理机制确保 cmd_fifo 非空时自动触发 AXI 事务启动。

\n\n

4.4 并发与冲突处理

\n

由于 AXI4-Lite 仅支持单次传输且所有事务串行执行，SPI2AXI Bridge 不支持多通道并发访问。读写操作通过 FSM 状态进行区分，且读/写数据的 FIFO 路径相互独立，因此不会产生读写冲突。AXI4-Lite 协议本身不支持独占访问（locked transaction）或原子操作，本设计遵循协议规范，不实现相关机制。

6. 时钟、复位与电源架构

\n

本章节定义 SPI2AXI Bridge 的时钟域划分、复位结构以及电源模式。正确的时钟域和复位设计是保证芯片功能正确性和可靠性的基础，特别是在涉及跨时钟域数据传输的场景下。

\n\n

6.1 时钟域概述

\n

SPI2AXI Bridge 包含两个完全异步的时钟域：spi_sclk 时钟域由外部 SPI 主机提供，最大频率为 50 MHz，覆盖 spi_slave_if、fsm_ctrl（SPI 侧）、csr_regfile 和 SPI 侧 CDC FIFO 逻辑；axi_aclk 时钟域由 SoC PLL 提供，频率待定（通常为 50~200 MHz），覆盖 axi_master_if、wrap_addr_gen 和 AXI 侧 CDC FIFO 逻辑。两个时钟域之间通过 dual-clock FIFO 实现异步数据传输，所有跨时钟域路径被标记为 false path，不参与静态时序分析 (STA)。

\n

SPI2AXI Bridge 采用两个独立的异步复位信号，分别对应 SPI 域和 AXI 域。spi_rst_n（低有效异步复位）复位 SPI 时钟域的所有时序逻辑，包括 SPI 接口逻辑和 SPI 侧寄存器；axi_areset_n（低有效异步复位）复位 AXI 时钟域的所有时序逻辑，包括 AXI 主接口和 AXI 侧寄存器。复位结构建议采用异步复位、同步释放（asynchronous assertion, synchronous de-assertion）的复位同步器设计方案，以避免复位释放时的亚稳态问题。CDC FIFO 需要两个时钟域都完成复位才能正确初始化。

\n\n

用例 1: 系统调试和配置接口 (System Debug & Configuration)

\n

外部调试主机通过 SPI 连接 SoC 的 SPI2AXI 接口，用于芯片调试阶段的寄存器读写和状态查询。每次操作的数据量为 4 字节 (32-bit)，访问对象为 SoC 内部配置空间或外设 CSR 寄存器。操作序列为：外部主机通过 SPI 发送操作码 + 地址 + 数据（写操作）或操作码 + 地址（读操作），SPI2AXI 解析命令并转换为 AXI 事务，读数据通过 SPI 返回给外部主机。该场景的关键要求是低延迟和可靠传输。

用例 2: 嵌入式系统固件更新 (Firmware Update)

\n

外部主机发送固件更新命令和固件数据（KB ~ MB 级别），通过 SPI2AXI 将数据写入 SoC 内部存储器。该场景下，外部主机首先配置 SPI2AXI 为 QSPI 模式 (4-wire) 以获得更高吞吐，并可根据需要使能 Wrap 功能。随后连续发送写命令 + 地址 + 数据，地址自动递增，固件数据通过 AXI 写入目标存储器。该场景的关键要求是高吞吐（利用 QSPI 的 4-wire 并行能力）和地址连续递增。

\n\n

用例 3: 芯片测试与验证 (Chip Test & Verification)

\n

测试台通过 SPI 访问芯片内部状态，进行寄存器校验和存储器扫描。测试台发送读命令 + 地址，等待 dummy cycles 后读取返回数据，验证内部寄存器/存储器值的正确性。Wrap 功能在连续地址扫描场景下特别有用，可以减少每次访问的地址开销。该场景的关键要求是寄存器访问的准确性和 Wrap 功能的可靠性。

\n\n

8.2 异常场景

\n

SPI2AXI Bridge 需要处理以下异常场景：AXI 总线超时（从设备返回 SLVERR 或 DECERR 错误响应）→ 错误标志位 (err flag) 置位；SPI 片选异常释放（片选在数据传输完成前被释放）→ 当前事务终止，FSM 返回 IDLE 状态；FIFO 溢出（写数据 FIFO 满时 SPI 仍在写入）→ 待补充处理方式；非法操作码（未定义的操作码）→ 待补充处理方式。这些异常场景的具体处理机制将在微架构设计 (LLD) 中进一步细化。

8.3 使用限制

\n

SPI2AXI Bridge 的使用受以下限制：仅支持 AXI4-Lite single transfer（burst 长度固定为 1）；不支持 AXI4-Full 的 burst、locked 事务、cached 或 protection unit 等高级特性；SPI 地址必须与 4 字节对齐 (AxSIZE = 2)；SPI 时钟频率不可超过 50 MHz；跨时钟域路径通过 dual-clock FIFO 保证，不依赖静态时序分析 (STA) 约束。

\n 370→

\n

1.2 一句话摘要

\n

SPI2AXI Bridge 是一个将 SPI 从设备接口 (SPI Slave) 转换为 AXI4-Lite 主设备接口 (AXI4-Lite Master) 的桥接 IP 核，允许外部 SPI 主机通过低引脚数串行接口访问 SoC 内部的 AXI 总线映射存储器和外设配置空间。该桥接器通过双时钟域 FIFO 实现可靠的跨时钟域数据传输（Clock Domain Crossing, CDC），并支持标准 SPI 与 Quad SPI (QSPI) 两种工作模式，为芯片调试、固件更新和系统配置提供灵活的高效通道。

\n\n

1.3 功能目标

\n

SPI2AXI Bridge 的核心功能目标包括以下五个方面：

\n
    \n
  • SPI 协议支持：支持标准 SPI (1-wire) 和 Quad SPI / QSPI (4-wire) 两种工作模式，通过控制状态寄存器（Control and Status Register, CSR）可配置切换，满足不同带宽需求的场景
    • \n
  • 协议转换：将 SPI 接收到的 8-bit 操作码和 32-bit 地址解码并转换为 AXI4-Lite 读写事务，实现串行协议到并行总线协议的无缝转换
    • \n
  • 跨时钟域传输：通过 dual-clock FIFO 实现 SPI 时钟域到 AXI 时钟域的可靠跨时钟域数据传输 (CDC)，采用格雷码 (Gray Code) 指针同步机制确保异步时钟域间的数据完整性和亚稳态防护
    • \n
  • 地址 Wrap 功能：支持可配置的地址 Wrap 功能，在 AXI4-Lite 单次传输约束下模拟循环地址访问，特别适用于 FIFO 轮询和连续地址扫描场景
    • \n
  • 可编程配置：提供寄存器文件 (CSR) 用于工作模式配置、状态查询和参数配置，所有寄存器均通过 SPI 操作码直接编址访问
    • \n

1.6 功能边界

\n

2.2 SPI 从设备接口

\n

SPI 从设备接口是 SPI2AXI Bridge 的串行数据入口，支持标准 SPI (1-wire) 和 Quad SPI (4-wire) 两种工作模式。接口同步于外部 SPI 主机提供的 spi_sclk 时钟，最高工作频率为 50 MHz。协议采用 SPI 模式 0 (CPOL=0, CPHA=0)，即串行时钟 (sclk) 空闲状态为低电平，数据在 sclk 上升沿采样、下降沿输出。数据传输顺序为 MSB first（最高有效位在前），与大多数 SPI 外设和微控制器的默认配置兼容。

\n\n

SPI 帧格式根据访问类型分为四种：内存写帧格式为 [Opcode (8-bit)] + [Address (32-bit)] + [Write Data (32-bit)]；内存读帧格式为 [Opcode (8-bit)] + [Address (32-bit)] + [Dummy Cycles (可编程)] + [Read Data (32-bit)]；寄存器写帧格式为 [Opcode (8-bit)] + [Write Data (32-bit)]；寄存器读帧格式为 [Opcode (8-bit)] + [Dummy Cycles] + [Read Data (32-bit)]。其中，Dummy Cycles 的数量可通过 CSR 中的 DUMMY_CFG 寄存器进行编程配置，实际等待周期数为配置值加 1，以确保在 AXI 读数据返回前 SPI 侧有足够的时间等待。

\n\n\n\n\n

2.3 AXI4-Lite 主设备接口

\n

AXI4-Lite 主设备接口是 SPI2AXI Bridge 连接 SoC 内部 AXI 总线矩阵的并行协议接口。本设计采用 AXI4-Lite 协议，它是 AXI4 协议的轻量级子集，不支持 burst 传输，固定突发长度 (AxLEN) 为 0（即 single transfer），突发大小 AxSIZE 固定为 2（32-bit / 4-byte 传输）。数据位宽固定为 32-bit，与 AXI4-Lite 协议规范一致。

\n\n

AXI4-Lite 接口包含五个独立的通道：写地址通道 (AW)、写数据通道 (W)、写响应通道 (B)、读地址通道 (AR) 和读数据通道 (R)。每个通道采用独立的 valid-ready 握手协议，通道间无顺序依赖。本设计使用 3-bit ID 宽度 (AWID/ARID/WID/RID/BID)，以满足 SoC 级 ID 追踪需求。AXI4-Lite 不支持 out-of-order 事务完成，所有事务按发起顺序完成。

\n\n

2.4 中断接口

\n

中断接口输出一个单比特电平中断信号 intr_o，该信号同步于 AXI 时钟域 (axi_aclk)。中断事件源主要包括 AXI 事务错误（如 SLVERR 或 DECERR 响应）等异常状况。中断清除机制采用读状态寄存器清除或写 1 清除（Write-1-to-Clear, W1C）方式，具体实现将在微架构设计 (LLD) 中详细定义。

3. 顶层架构框图

\n

本章节从系统层面展示 SPI2AXI Bridge 的内部模块划分和互联关系。架构框图清晰地呈现了两个时钟域（SPI 时钟域和 AXI 时钟域）的划分、六个核心子模块的职责边界以及模块间的关键数据通道。整体架构遵循 CDC FIFO 隔离设计原则，将 SPI 协议处理逻辑和 AXI 协议处理逻辑分别置于各自的时钟域中，仅在必要时通过 dual-clock FIFO 进行数据交换。这种架构设计最大限度地减少了跨时钟域路径，降低了 CDC 验证的复杂度。

\n\n\n\n

3.1 模块内部结构

\n\n

SPI2AXI Bridge 的内部架构可划分为 SPI 时钟域和 AXI 时钟域两大区域，通过三路独立的 dual-clock FIFO（命令 FIFO cmd_fifo、写数据 FIFO wdata_fifo、读数据 FIFO rdata_fifo）实现跨时钟域隔离和数据传输。

\n\n

SPI 时钟域 (spi_sclk) 包含以下逻辑单元：spi_slave_if（SPI 串行接口收发与命令解析）、fsm_ctrl（主状态机，控制 SPI 到 AXI 的整个事务流程）以及 csr_regfile（配置/状态寄存器文件）。SPI 时钟域的所有时序逻辑均同步于外部 SPI 主机提供的 spi_sclk，该时钟频率最高 50 MHz，由外部 SPI 主机控制，与 SoC 内部时钟完全异步。

\n\n

AXI 时钟域 (axi_aclk) 包含以下逻辑单元：axi_master_if（AXI4-Lite 主接口协议处理，负责 AW/W/B/AR/R 五个通道的握手协议）和 wrap_addr_gen（Wrap 地址生成和管理模块，根据 CSR 配置实现地址自增与回绕逻辑）。AXI 时钟域的所有时序逻辑均同步于 SoC 提供的 axi_aclk，工作频率待定（通常为 50~200 MHz）。

\n\n

顶层还包含 Clock / Reset / CDC Synchronizer 模块，负责两个时钟域的复位同步（异步复位同步释放）和全局时钟管理。

4. 数据流与控制流

\n

本章节详细描述 SPI2AXI Bridge 在读写操作中的完整数据流路径和控制流状态迁移。数据流部分按写操作和读操作分别阐述从 SPI 引脚到 AXI 总线的串行→并行→协议封装的完整链条。控制流部分通过主状态机 (FSM) 的状态迁移图展示 SPI 命令解析到 AXI 事务完成的完整流程，并分析反压传播路径和并发冲突处理策略。

\n\n

4.1 数据流路径

\n\n

写操作数据流路径如下：SPI 引脚接收串行数据 → spi_slave_if 进行串并转换（serial-to-parallel conversion）→ 命令/地址写入 cmd_fifo，写数据写入 wdata_fifo → 经过 CDC 同步到 AXI 时钟域 → axi_master_if 从 FIFO 读取命令和数据 → 发起 AXI4-Lite 写事务 (AW 通道发送地址 → W 通道发送数据 → B 通道接收响应)。

\n\n

读操作数据流路径如下：SPI 引脚接收串行数据 → spi_slave_if 串并转换 → 命令/地址写入 cmd_fifo → 经过 CDC 同步到 AXI 时钟域 → 在 SPI 域插入可编程 dummy cycles（等待 AXI 读数据返回） → axi_master_if 从 cmd_fifo 获取地址 → 发起 AXI4-Lite 读事务 (AR 通道发送地址 → R 通道接收数据) → 读数据写入 rdata_fifo → 经过 CDC 同步回 SPI 时钟域 → spi_slave_if 从 rdata_fifo 读取数据 → 并串转换 (parallel-to-serial conversion) → 通过 spi_sdo 串行发送给外部 SPI 主机。

\n\n

\n

    2. \n
  2. SPI 数据接收: spi_slave_if 接收写数据 (32-bit)，完成写入数据的串并转换
    3. \n
  3. CDC 传输: 命令/地址写入 cmd_fifo，写数据写入 wdata_fifo，通过格雷码指针同步到 AXI 时钟域
    4. \n
  4. AXI 写事务: axi_master_if 从 cmd_fifo 获取地址，从 wdata_fifo 获取数据，发起 AXI 写事务
    5. \n

    6. \n

4.2 控制流 — 主状态机

\n\n

主状态机 (FSM) 位于 SPI 时钟域，控制从 SPI 命令接收到 AXI 事务完成的完整流程。FSM 共包含 8 个状态：IDLE（空闲等待）、OPCODE（接收 8-bit 操作码）、ADDR（接收 32-bit 地址）、DUMMY（读路径上的虚拟等待周期）、DATA（32-bit 数据传输）、AXI_WR（AXI 写事务执行）、AXI_RD（AXI 读事务执行）和 DONE（事务完成，返回 IDLE）。

\n\n

FSM 的正常状态转换路径为：IDLE → OPCODE → ADDR → DUMMY (仅读路径) → DATA → AXI_WR/AXI_RD → DONE → IDLE。寄存器访问模式下，ADDR 状态会被跳过（因为寄存器地址直接由操作码编码）。

4.3 反压与流控策略

\n

SPI2AXI Bridge 在数据流路径上实现了端到端的反压传播机制 (Backpressure Propagation)，确保在 AXI 总线拥塞或 FIFO 满溢时能够妥善处理。写操作的反压传播路径为：AXI 总线反压（从设备未就绪）→ axi_master_if 等待握手完成 → wdata_fifo 满（无法写入新数据）→ SPI 域暂停数据发送。读操作的反压传播路径为：rdata_fifo 空（读数据尚未返回）→ SPI 域等待 → axi_master_if 等待 R 通道响应。FIFO 的水位管理机制确保 cmd_fifo 非空时自动触发 AXI 事务启动，实现流水线化的操作流程。

\n\n

4.4 并发与冲突处理

\n

由于 AXI4-Lite 仅支持单次传输且所有事务串行执行，SPI2AXI Bridge 不支持多通道并发访问。读写操作通过 FSM 状态进行区分，且读/写数据的 FIFO 路径相互独立，因此不会产生读写冲突。AXI4-Lite 协议本身不支持独占访问（locked transaction）或原子操作，本设计遵循协议规范，不实现相关机制。所有事务严格按照 SPI 命令接收的先后顺序串行转换为 AXI 事务。

6. 时钟、复位与电源架构

\n

本章节定义 SPI2AXI Bridge 的时钟域划分、复位结构以及电源模式。正确的时钟域和复位设计是保证芯片功能正确性和可靠性的基础，特别是在涉及跨时钟域数据传输 (CDC) 的场景下。本设计采用双时钟域异步架构，所有跨时钟域路径通过 dual-clock FIFO 和格雷码同步器处理。

\n\n

6.1 时钟域概述

\n

SPI2AXI Bridge 包含两个完全异步的时钟域：spi_sclk 时钟域由外部 SPI 主机提供，最大频率为 50 MHz，覆盖 spi_slave_if、fsm_ctrl（SPI 侧）、csr_regfile 和 SPI 侧 CDC FIFO 逻辑；axi_aclk 时钟域由 SoC PLL 提供，频率待定（通常为 50~200 MHz），覆盖 axi_master_if、wrap_addr_gen 和 AXI 侧 CDC FIFO 逻辑。两个时钟域之间通过 dual-clock FIFO 实现异步数据传输，所有跨时钟域路径在 STA 中被标记为 false path，不参与静态时序分析。

\n

SPI2AXI Bridge 采用两个独立的异步复位信号，分别对应 SPI 域和 AXI 域。spi_rst_n（低有效异步复位）复位 SPI 时钟域的所有时序逻辑，包括 SPI 接口逻辑和 SPI 侧寄存器；axi_areset_n（低有效异步复位）复位 AXI 时钟域的所有时序逻辑，包括 AXI 主接口和 AXI 侧寄存器。复位结构建议采用异步复位、同步释放 (asynchronous assertion, synchronous de-assertion) 的复位同步器设计方案，以避免复位释放时的亚稳态问题。CDC FIFO 需要两个时钟域都完成复位才能正确初始化。

本章节描述 SPI2AXI Bridge 的典型应用用例和异常场景处理。通过对实际使用场景的分析，可以更好地理解 IP 的功能需求和性能要求，也为后续的验证方案设计提供输入。

8.1 典型用例

\n\n

用例 1: 系统调试和配置接口 (System Debug & Configuration)

\n

外部调试主机通过 SPI 连接 SoC 的 SPI2AXI 接口，用于芯片调试阶段的寄存器读写和状态查询。每次操作的数据量为 4 字节 (32-bit)，访问对象为 SoC 内部配置空间或外设 CSR 寄存器。操作序列为：外部主机通过 SPI 发送操作码 + 地址 + 数据（写操作）或操作码 + 地址（读操作），SPI2AXI 解析命令并转换为 AXI 事务，读数据通过 SPI 返回给外部主机。该场景的关键要求是低延迟和可靠传输。通过 SPI 的 4 线引脚（数据 + 时钟 + 片选），外部调试器可以完全访问 SoC 内部地址空间，不需要额外的 JTAG 接口。

\n\n

用例 2: 嵌入式系统固件更新 (Firmware Update)

\n

外部主机发送固件更新命令和固件数据（KB ~ MB 级别），通过 SPI2AXI 将数据写入 SoC 内部存储器。该场景下，外部主机首先配置 SPI2AXI 为 QSPI 模式 (4-wire) 以获得更高吞吐（相较于标准 SPI 提升 4 倍），并可根据需要使能 Wrap 功能。随后连续发送写命令 + 地址 + 数据，地址自动递增，固件数据通过 AXI 写入目标存储器。该场景的关键要求是高吞吐（利用 QSPI 的 4-wire 并行能力）和地址连续递增。

\n\n

用例 3: 芯片测试与验证 (Chip Test & Verification)

\n

测试台通过 SPI 访问芯片内部状态，进行寄存器校验和存储器扫描。测试台发送读命令 + 地址，等待 dummy cycles 后读取返回数据，验证内部寄存器/存储器值的正确性。Wrap 功能在连续地址扫描场景下特别有用，可以减少每次访问的地址开销。该场景的关键要求是寄存器访问的准确性和 Wrap 功能的可靠性。

\n\n

8.2 异常场景

\n

SPI2AXI Bridge 需要处理以下异常场景：AXI 总线超时（从设备返回 SLVERR 或 DECERR 错误响应）→ 错误标志位 (err flag) 置位，中断输出有效；SPI 片选异常释放（片选在数据传输完成前被释放）→ 当前事务终止，FSM 返回 IDLE 状态，未完成的数据被丢弃；FIFO 溢出（写数据 FIFO 满时 SPI 仍在写入）→ 待补充处理方式；非法操作码（未定义的操作码）→ 待补充处理方式。这些异常场景的具体处理机制将在微架构设计 (LLD) 中进一步细化。

\n\n

8.3 使用限制

\n

SPI2AXI Bridge 的使用受以下限制：仅支持 AXI4-Lite single transfer (burst 长度固定为 1)；不支持 AXI4-Full 的 burst、locked 事务、cached 或 protection unit 等高级特性；SPI 地址必须与 4 字节对齐 (AxSIZE = 2)，即地址最低 2 位必须为 0；SPI 时钟频率不可超过 50 MHz；跨时钟域路径通过 dual-clock FIFO 保证，不依赖静态时序分析 (STA) 进行路径时序收敛。

1.2 One-Sentence Summary

\n

The SPI2AXI Bridge is a bridge IP core that converts a SPI Slave interface into an AXI4-Lite Master interface, allowing an external SPI master to access SoC-internal AXI bus-mapped memory and peripheral configuration space through a low-pin-count serial interface. The bridge achieves reliable cross-clock domain data transfer (Clock Domain Crossing, CDC) via dual-clock FIFOs and supports both Standard SPI and Quad SPI (QSPI) operating modes, providing a flexible and efficient channel for chip debug, firmware update, and system configuration.

\n\n

1.3 Functional Targets

\n

The core functional targets of the SPI2AXI Bridge include the following five aspects:

\n
    \n
  • SPI Protocol Support: Standard SPI (1-wire) and Quad SPI / QSPI (4-wire) modes, switchable via CSR configuration to meet different bandwidth requirements
    • \n
  • Protocol Conversion: Decode 8-bit opcode and 32-bit address from SPI and convert into AXI4-Lite read/write transactions, achieving seamless serial-to-parallel protocol conversion
    • \n
  • Cross-Clock Domain Transfer: Reliable CDC data transfer via dual-clock FIFO with Gray code pointer synchronization for data integrity and metastability protection
    • \n
  • Address Wrap: Configurable address wrap function for cyclic address access under AXI4-Lite single transfer constraints, ideal for FIFO polling and sequential address scanning
    • \n
  • Programmable Configuration: CSR register file for mode configuration, status query, and parameter setup, all directly addressable via SPI opcodes
    • \n

2.2 SPI Slave Interface

\n

The SPI slave interface is the serial data entry point for the SPI2AXI Bridge, supporting both Standard SPI (1-wire) and Quad SPI (4-wire) modes. The interface is synchronous to the external spi_sclk with a maximum operating frequency of 50 MHz. The protocol uses SPI Mode 0 (CPOL=0, CPHA=0), where sclk idles low and data is sampled on the rising edge. Data transmission follows MSB-first ordering.

\n\n

SPI frame formats are classified into four types: Memory Write [Opcode 8-bit + Address 32-bit + Write Data 32-bit], Memory Read [Opcode + Address + Dummy Cycles + Read Data 32-bit], Register Write [Opcode + Write Data 32-bit], and Register Read [Opcode + Dummy Cycles + Read Data 32-bit]. Dummy cycle count is programmable via the DUMMY_CFG CSR register, with actual wait cycles equal to the configured value plus one.

\n\n\n\n\n

2.3 AXI4-Lite Master Interface

\n

The AXI4-Lite master interface connects the SPI2AXI Bridge to the SoC internal AXI bus matrix. The design uses the AXI4-Lite protocol, a lightweight subset of AXI4, with fixed burst length of 1 (AxLEN=0, single transfer), burst size AxSIZE=2 (32-bit/4-byte), and 32-bit data width. The interface comprises five independent channels: write address (AW), write data (W), write response (B), read address (AR), and read data (R), each with independent valid-ready handshaking. A 3-bit ID width is used for SoC-level ID tracking.

\n\n

2.4 Interrupt Interface

\n

The interrupt interface outputs a single-bit level interrupt signal intr_o, synchronized to the AXI clock domain. Interrupt sources include AXI transaction errors (SLVERR/DECERR). The interrupt clear mechanism uses read-to-clear or Write-1-to-Clear (W1C) via the STATUS register.

4. Data Flow & Control Flow

\n

This section describes the complete data flow paths and control flow state transitions for both read and write operations.

\n\n

4.1 Data Flow Paths

\n\n

Write Data Path: SPI Pins (serial) → spi_slave_if (serial-to-parallel conversion) → cmd_fifo + wdata_fifo → CDC sync to AXI domain → axi_master_if (protocol encapsulation) → AXI4-Lite write transaction (AW → W → B).

\n\n

Read Data Path: SPI Pins (serial) → spi_slave_if → cmd_fifo → CDC sync → axi_master_if → AXI4-Lite read transaction (AR → R) → rdata_fifo → CDC sync back to SPI domain → spi_slave_if (parallel-to-serial) → SPI Pins (serial).

\n\n

Write Operation Steps:

\n
    \n
  1. SPI command reception: 8-bit opcode + 32-bit address via spi_sdi, MSB first
    2. \n
  2. SPI data reception: 32-bit write data, serial-to-parallel conversion
    3. \n
  3. CDC transfer: command/address to cmd_fifo, write data to wdata_fifo, Gray code pointer synchronization
    4. \n
  4. AXI write transaction: fetch address from cmd_fifo, data from wdata_fifo
    5. \n
  5. AXI handshake: AW channel (address) → W channel (data) → B channel (response)
    6. \n
  6. Completion: status update, FSM returns to IDLE
    7. \n
\n\n\n\n\n

4.2 Control Flow — Main State Machine

\n

The main FSM resides in the SPI clock domain and controls the complete flow from SPI command reception to AXI transaction completion. The FSM has 8 states: IDLE, OPCODE, ADDR, DUMMY, DATA, AXI_WR/AXI_RD, and DONE.

4.3 Backpressure & Flow Control

\n

The SPI2AXI Bridge implements end-to-end backpressure propagation. Write backpressure: AXI bus backpressure → axi_master_if waits → wdata_fifo full → SPI domain pauses transmission. Read backpressure: rdata_fifo empty → SPI domain waits → axi_master_if waits for R channel response.

\n\n

4.4 Concurrency & Conflict Handling

\n

Since AXI4-Lite supports only single transfers and all transactions execute serially, the SPI2AXI Bridge does not support multi-channel concurrent access. Read/write operations are distinguished by FSM state, with independent FIFO paths preventing read-write conflicts.

8. Application Scenarios

\n

This section describes typical use cases and exception scenarios for the SPI2AXI Bridge, providing input for verification planning.

8.1 Typical Use Cases

\n\n

Use Case 1: System Debug & Configuration
\nAn external debug host connects via SPI to access SoC internal registers and memory. Single 4-byte (32-bit) register read/write operations. The debugger sends opcode + address + data (write) or opcode + address (read), and the SPI2AXI bridge converts these to AXI transactions. Key requirement: low latency and reliable transmission.

\n\n

Use Case 2: Firmware Update
\nAn external host sends firmware data (KB to MB range) in QSPI mode (4-wire) for high throughput. The host configures the bridge to QSPI mode, enables wrap function as needed, and sends continuous write commands with auto-incrementing addresses. Key requirement: high throughput and sequential address access.

\n\n

Use Case 3: Chip Test & Verification
\nTest equipment accesses chip internal state via SPI for production testing. Register access with wrap functionality enables efficient memory scanning. Key requirement: access accuracy and wrap reliability.
