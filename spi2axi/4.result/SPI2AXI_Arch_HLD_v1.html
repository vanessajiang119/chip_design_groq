1.2 一句话摘要 (One-Line Summary)

SPI2AXI Bridge IP 是一个将 SPI 从设备（Slave）接口转换为 AXI4-Lite 主设备（Master）接口的桥接模块，允许外部 SPI 主机通过标准的 SPI 或 Quad-SPI（QSPI）总线协议，无需 SoC 内部 CPU 干预即可直接访问 AXI 总线上的存储器映射寄存器空间和外设，广泛用于 SoC 的系统调试、配置管理、固件更新以及芯片测试验证等场景。

1.3 功能目标 (Functional Goals)

  • 支持标准 SPI 和 Quad-SPI (QSPI) 双模协议: 提供 1 线标准 SPI 和 4 线 QSPI 两种工作模式，通过 SPI 侧配置寄存器动态切换数据线宽度，适应不同引脚数和带宽需求的系统应用场景。
  • 将 SPI 总线命令透明转换为 AXI4-Lite 总线事务: 解析 SPI 主机发送的 8-bit 操作码和 32-bit 地址，生成对应的 AXI4-Lite 写地址/写数据/读地址通道事务，实现 SPI 协议到 AXI 协议的完整语义映射。
  • 实现 SPI 时钟域到 AXI 时钟域的可靠跨时钟域数据传输: 采用双时钟异步 FIFO（dual-clock async FIFO）架构，在 SPI 时钟域（最高 50MHz）和 AXI 系统时钟域之间提供具有格雷码指针同步的可靠 CDC 传输通道。
  • 支持可配置的地址环绕（Wrap）访问模式: 提供软件可编程的地址环绕功能，Wrap=N 时以 N 个 32-bit 字（4×N 字节）为窗口进行地址回绕访问。
  • 提供 SPI 侧配置寄存器和状态反馈机制: 通过 SPI 操作码编码的寄存器地址空间，支持 SPI 工作模式选择、传输状态查询、Wrap 参数配置等功能。

1.6 功能边界 (Scope)

  • 范围内 (In-Scope): SPI 标准模式（1-line）和 Quad-SPI 模式（4-line）的读写操作
  • 范围内: AXI4-Lite 五通道（AW/W/B/AR/R）协议转换与握手管理
  • 范围内: SPI 时钟域与 AXI 时钟域之间的异步 FIFO CDC 同步
  • 范围内: 可配置地址环绕（Wrap）功能的地址计算与边界检测
  • 范围外 (Out-of-Scope): AXI4-Full 协议支持
  • 范围外: SPI 主设备（Master）模式 — 仅作为 SPI Slave 工作
  • 范围外: 多 SPI 片选支持
  • 范围外: 中断输出机制

2.2 配置接口 (Configuration Interface)

  • 协议: SPI Slave（标准 SPI / Quad-SPI）
  • 最大时钟频率: 50 MHz (SPI sclk)
  • 地址空间: 8-bit 操作码编码的寄存器空间 + 32-bit 地址的存储器空间
  • 传输时序: SPI 模式 0（CPOL=0, CPHA=0），上升沿采样，下降沿更新
  • 写操作帧格式: OPCODE(8-bit) + ADDR(32-bit) + DATA(32-bit) — 标准 SPI 共需 72 SCLK
  • 读操作帧格式: OPCODE(8-bit) + ADDR(32-bit) + DUMMY(N+1) + DATA(32-bit)

2.3 数据面接口 (Data Interface)

  • 协议: AXI4-Lite (Master)
  • 数据位宽: 32-bit
  • 通道数: 5 个独立通道（AW, W, B, AR, R）
  • 突发长度: 1 beat (AxLEN=0)
  • 关键时序: AW 和 W 通道独立 valid；AR 通道发起读地址；等待 ready 握手

2.4 中断接口 (Interrupt Interface)

本模块不包含专用中断输出。SPI 主机通过读取 SPI 侧状态寄存器获取传输完成/错误状态，采用轮询方式。适用于系统调试和配置场景，避免跨时钟域和跨芯片边界的中断同步复杂性。

4.1 数据流路径 (Data Path)

SPI2AXI Bridge 的核心数据流分为写数据流和读数据流两条独立路径。

写数据流路径 (Write Data Path):

[SPI Master] → spi_sdi → [spi_slave: 串并转换] → [cmd_decoder: 命令解析]

                                                        ▼
                                              ┌─────────────────────┐
                                              │   async_fifo:       │
                                              │  Write Cmd FIFO     │◄── 写命令

                                              └─────────┬───────────┘
                                                        │ (CDC: spi_sclk → axi_aclk)

                                              │  B Channel          │◄──── AXI Bus

4.2 控制流 (Control Flow)

控制核心是 cmd_decoder 的命令解码 FSM，管理从 SPI 命令接收到 AXI 事务完成的完整流程。

stateDiagram-v2 [*] --> IDLE IDLE --> OPCODE: spi_cs_n = 0 OPCODE --> ADDR: 8-bit done ADDR --> DUMMY: read opcode ADDR --> DATA: write opcode DUMMY --> DATA: dummy done DATA --> RESPONSE: 32-bit done RESPONSE --> IDLE: resp done IDLE --> IDLE: spi_cs_n = 1 OPCODE --> IDLE: spi_cs_n = 1 (abort) ADDR --> IDLE: spi_cs_n = 1 (abort) DUMMY --> IDLE: spi_cs_n = 1 (abort) DATA --> IDLE: spi_cs_n = 1 (abort) RESPONSE --> IDLE: spi_cs_n = 1 (abort)
状态 (State)描述 (Description)
IDLE空闲等待，SPI Slave 已准备好接收新传输
OPCODE接收 8-bit 操作码，MSB 先传输
ADDR接收 32-bit 目标地址
DUMMY插入可编程数量的虚拟 SCLK 周期，等待 AXI 读数据返回
DATA数据传输：接收写数据或发送读数据
RESPONSEAXI 写响应/读完成确认

4.3 反压与流控 (Backpressure)

反压传播链：AXI Slave 响应慢 → axi_master 握手延迟 → async_fifo 满 → cmd_decoder 暂停 → spi_slave 保持。FIFO 深度建议配置为 4~8 个条目。建议在 cmd_decoder 中添加 AXI 响应超时检测（1024 个 AXI 时钟周期阈值）。

第8章 应用场景 (Application Scenarios)

8.1 典型用例 (Use Cases)

用例 1: SoC 系统调试和配置接口 (SoC Debug & Config)

外部调试主机通过 SPI 总线访问 SoC 内部寄存器空间，实现非侵入式在线调试。

  • 触发条件: 外部调试主机通过 SPI 发起 AXI 寄存器访问请求
  • 操作序列: 配置 SPI 模式 → WRITE 操作写入目标地址 → READ 操作回读验证
  • 关键要求: 读延迟可控（DUMMY_CYCLES 参数调节），延迟确定性高

用例 2: 嵌入式系统固件批量更新 (Firmware Update)

外部编程器通过 QSPI 4-line 高速模式批量写入固件镜像。

  • 触发条件: 系统进入固件更新模式
  • 操作序列: 配置 QSPI 模式 → 连续写帧 → 状态查询确认
  • 关键要求: QSPI 峰值吞吐 25 MB/s

用例 3: 芯片 ATE 测试配置通路 (ATE Test Config)

8.2 异常场景 (Exception Scenarios)

  • 片选异常拉高: FSM 立即回到 IDLE，丢弃当前不完整帧
  • AXI Slave 错误: SLVERR/DECERR 记录到状态寄存器
  • FIFO 满: cmd_decoder 暂停写入，SPI 侧保持当前状态
  • 非法操作码: 记录错误标志，终止处理
  • 软复位: CTRL 寄存器 soft_rst=1 触发全局复位
