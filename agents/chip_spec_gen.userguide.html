简介

Chip Spec Gen Agent（芯片设计规格生成智能体）是一个面向 SoC / ASIC / IP 芯片设计领域的专业文档自动化生成工具。它接收用户提供的芯片设计源材料（HLD 文档、技术手册、设计说明、架构图片等），通过自动化的多阶段流水线处理，生成结构化的、中英双语的、包含专业架构图的芯片设计规格文档。

该 Agent 适用于以下场景：

  • 将技术手册或设计笔记整理为结构化 HLD/LLD 规格文档
  • 将设计源材料按章节拆分后重新组织为标准输出格式
  • 批量生成包含架构图的设计评审材料
  • 文档标准化迁移（从非结构化到专业结构化的转换）

快速开始

使用 Chip Spec Gen Agent 只需 5 步：

1
提交源文档

将你的芯片设计源文档提交给 Agent。支持的格式包括：markdown（.md）、图片（.png/.jpg）、WORD（.docx）、PDF（.pdf）等。同时说明你需要的输出文档类型和章节要求。

2
自动规划

Agent 自动解析源文档，转换为内部 md 格式，并生成 planning.yml 配置文件。你可以在执行前审查规划参数（如迭代次数、输出章节结构）并进行调整。

3
切片与整理

Agent 自动将源文档按章节切片提取，然后重新组织为输出要求的章节结构。检查内容完整性，对缺失内容进行多轮迭代补全。

4
审查进度

Agent 在整理阶段会输出阶段性分析报告，显示章节覆盖状态和完整度。你可以查看缺失内容清单，决定是否继续迭代或直接进入输出阶段。

5
获取最终报告

迭代完成后，Agent 生成三种格式的输出：NVIDIA 白色主题的 HTML 规格文档、可编辑的 .drawio 架构图文件、以及辅助 markdown 文件。所有文件保存在 4.result/ 目录中。

💡 提示： 你可以在任意阶段介入审查 — 查看规划配置、检查切片质量、或要求补充特定章节内容。Agent 的设计理念是"人机协作"，不是完全黑盒。

工作流程详解

Phase 1: Planning — 规划阶段

规划阶段是流程的起点。用户提交源文档后，planning 子 Agent 会：

  • 创建四阶段目录结构（1.planning/, 2.slice/, 3.working/, 4.result/）
  • 读取并解析用户输入的源文件（支持多格式：md、图片、WORD、PDF 等）
  • 将源文件统一转换为 markdown 格式，写入 1.planning/
  • 生成 planning.yml 配置文件，包含迭代次数和输出需求

你可以通过 planning.yml 控制：迭代轮数（max_iterations）、输出章节结构、需要的图表类型。如果输入源文件就是 md 格式，规划阶段会快速跳过格式转换环节。

Phase 2: Slice — 切片阶段

切片阶段将源文档按章节拆分为独立的内容块。slice 子 Agent 会：

  • 识别文档的一级/二级标题作为章节边界
  • 每章提取为独立的 slice-N-章节名.md 文件
  • 提取图片到 2.slice/images/ 目录，附加上下文标签
  • 保留表格的行列结构和代码块格式

此阶段只提取不修改，确保原始技术信息完整无损。切片后的材料是后续整理阶段的基础。

Phase 3: Working — 整理阶段（核心迭代）

整理阶段是流程的分析决策中心。working 子 Agent 执行"重组→检查→迭代"的核心循环：

  • 重组：按输出章节要求排列文字和图片，轻度润色确保术语一致
  • 检查：逐章检查文字和图表是否完备
  • 迭代：缺失内容 → 回 Phase 2 补充；完整 → 进入 Phase 4

每一轮迭代输出分析报告到 3.working/。报告显示每章的覆盖状态（✅ 完成 / ❌ 缺失）、完整度百分比。缺失的内容如果没有在源文档中找到，则标记为"待补充"，不会凭空编造。

Phase 4: Result — 输出阶段

输出阶段将整理后的内容生成最终交付物。workingresult 子 Agent 会：

  • 使用 html_chip_design_spec skill 生成 NVIDIA 白色主题的 HTML 规格文档
  • 使用 drawio_chip_diagram skill 生成芯片架构图
  • 每组图表以 SVG + mxGraphModel XML + .drawio 三种形式存在
  • 生成辅助 markdown 文件（含内嵌图片引用）

所有文件写入 4.result/ 目录，文件名带时间戳（YYYYMMDD-HHMM）以区分版本。

配置说明

planning.yml 是 Chip Spec Gen Agent 的核心配置文件，位于 1.planning/ 目录。你可以在规划阶段审查和修改此文件。

topic: <文档主题>
max_iterations: 5   # 迭代轮数，可修改
source_files:
  - source-doc-01.md
  - source-doc-02.md
output_requirements:
  chapters:
    - 1. 功能概述
    - 2. 系统架构
    - 3. 接口定义
    - 4. 特性列表
    - 5. 性能指标
  formats:
    - html           # NVIDIA 白色主题 HTML
    - drawio         # Draw.io 架构图
    - markdown       # 辅助 md 文档
FieldDefaultDescription
max_iterations5最大迭代轮数。每轮 working 发回 slice 寻找补充材料。达到此值后即使有缺失也进入 Phase 4
output_requirements.chapters—输出文档的章节列表。working 子 Agent 按此列表组织内容和检查完整性
output_requirements.formats[html, drawio, markdown]输出文件格式。目前固定支持 html + drawio + md 三种
调整建议： 如果你希望 Agent 更全面地搜索源文档中的内容，可以增大 max_iterations（例如 8）。如果源文档结构清晰、内容明确，5 轮通常足够。

最佳实践

✅ 推荐的输入方式
  • 源文档使用 markdown 格式，结构清晰（H1/H2 分层）
  • 源文档中明确章节标题和图片引用
  • 在最大迭代次数内让 Agent 自动迭代补全
  • 审查 planning.yml 中的 output_requirements 是否符合预期
  • 整理阶段检查完整性报告，决定是否继续迭代
❌ 避免的操作
  • 输入格式混乱、标题层次不明确的文档
  • 期望 Agent 自动发现源文档中不存在的内容
  • 跳过 planning.yml 审查直接运行完整流程
  • 同时提交大量不相关的文档材料
  • 在迭代补全阶段随意中断流程
综合建议： Chip Spec Gen Agent 最适合的场景是你有一些"半成品"素材（设计笔记、技术手册初稿、架构白板图、HLD 草稿），希望它们变成格式规范、语言专业、带架构图的完整设计规格文档。Agent 不会像 design_research 那样去搜索互联网补充知识——它只基于你给的材料工作。
