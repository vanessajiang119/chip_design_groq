1. 系统概览

Chip Spec Gen Agent（芯片设计规格生成智能体）是一个面向 SoC / ASIC / IP 芯片设计领域的专业文档自动化生成系统。该 Agent 的核心使命是：接收用户提供的芯片设计源材料（如 HLD 文档、技术手册、设计说明、架构图片等），通过 Planning（规划）→ Slice（切片）→ Working（整理）→ WorkingResult（输出） 四个阶段的多轮迭代，自动完成从原始材料解析到结构化芯片设计规格文档的完整生成流程。

系统采用 Orchestrator + Sub-Agent 架构模式，由一个主协调器（chip_spec_gen）和四个专用子 Agent 组成。协调器负责任务分发、状态管理和迭代控制；四个子 Agent 分别对应文档生成流程的四个阶段，形成完整的数据处理流水线。各子 Agent 之间通过标准化的文件系统目录进行数据交换，降低耦合度，支持用户在工作流任意阶段介入审查和调整方向。

系统以 1.planning/planning.yml 作为迭代控制中枢，默认支持最多 3 轮 内容补全迭代（用户可配置）。每一轮迭代中，slice 子 Agent 从源文档中按章节提取文字、图片、表格等原始材料；working 子 Agent 对切片数据进行分析整理，检查各章节的内容完整性，并决定是否需要进入下一轮 slice 迭代寻找补充材料。当所有章节内容完备或达到最大迭代次数后，workingresult 子 Agent 调用 html_chip_design_spec 和 drawio_chip_diagram 技能生成包含架构图的专业 HTML 芯片规格文档和配套的 .drawio 源文件。

该系统方案特别适用于芯片设计领域的技术规格文档整理、设计文档标准化迁移、各阶段设计评审材料准备、专利文档编制等需要从原始设计材料中提取并重构结构化文档的场景，能够显著减少芯片设计工程师在文档规范化方面的手动工作量。系统的多轮迭代补全机制模仿了人类工程师"粗读→提取→组织→查漏补缺"的自然文档撰写过程，确保最终输出文档的完整性和专业性。

2. 架构设计

Chip Spec Gen Agent 采用 Orchestrator（协调器）+ Sub-Agent（子代理） 的架构模式。主协调器负责任务分发、生命周期管理和迭代控制逻辑，四个专用子 Agent 各自独立执行一个文档处理环节。子 Agent 之间通过约定的文件系统目录进行数据交换，不直接通信，确保系统的可扩展性和可维护性。

架构的核心理念是单一职责：每个子 Agent 只关注自己阶段的核心任务，降低耦合度。迭代控制逻辑由 orchestrator 根据 working 子 Agent 输出的完整性评估信号和 planning.yml 中的 max_iterations 参数共同决定。各子 Agent 的配置定义在 agents/config/ 目录下，支持独立维护和版本管理。

3. 子 Agent 定义

Chip Spec Gen Agent 系统由 1 个主协调器和 4 个子 Agent 组成。下表概述各子 Agent 的核心配置和职责，后续小节逐个详述。

3.1 chip_spec_gen.planning — 规划子 Agent

planning 子 Agent 是整个流程的入口，负责处理用户输入的原始芯片设计源材料。其核心任务是对输入文件进行解析和格式转换，将各种格式（WORD、PDF、markdown、图片等）统一转换为内部 md 格式，并生成规划文件。

planning 子 Agent 会创建标准的四阶段目录结构（1.planning/, 2.slice/, 3.working/, 4.result/），并将解析后的源文件和规划文档输出到 1.planning/ 目录。其中 planning.yml 是核心配置文件，包含 max_iterations（默认 5）、source_files（源文件列表）、output_requirements（输出章节需求和格式要求）等关键字段，用户可以在该文件中调整迭代次数和输出范围。

配置定义在 agents/config/chip_spec_gen.planning.md。该子 Agent 专注于数据采集和需求分析阶段，不涉及内容提取和组织工作，遵循单一职责原则。

3.2 chip_spec_gen.slice — 切片子 Agent

slice 子 Agent 负责从 1.planning/ 中的源文档中按章节提取原始材料。其核心任务包括：识别文档的自然章节划分（基于一级/二级标题），将每章提取为独立的 md 文件，提取并标记图片引用，保留表格结构和代码块格式。

切片阶段的关键原则是只提取不修改。slice 子 Agent 不润色、不重组、不删减内容，确保原始技术信息的完整性。每张图片都附带上下文标签（描述它在文档中的位置和作用），每个表格都保留行列结构和数据内容。切片后的文件写入 2.slice/ 目录，以 slice-01-introduction.md 格式命名。如果源文档包含图片文件，切片 agent 会将其复制到 2.slice/images/ 子目录中。

配置定义在 agents/config/chip_spec_gen.slice.md。该子 Agent 在迭代过程中可能被 working 子 Agent 要求返回继续提取特定章节的补充材料。

3.3 chip_spec_gen.working — 整理子 Agent

working 子 Agent 是流程的分析决策中心。它读取 2.slice/ 中的切片数据，根据 1.planning/planning.yml 中的 output_requirements 确定最终输出文档的章节结构，然后重新组织内容并检查完整性。

working 的核心工作模式是"重组 + 检查 + 迭代"。首先，它按照输出章节要求排列文字和图片，对已有内容做轻度润色（语法修正、术语一致性调整），并标注每节内容的来源切片文件。然后，逐章检查是否每个章节都有对应的文字描述和必要的图表。如果发现缺失内容且当前迭代次数未达到 max_iterations，则返回 slice 阶段要求补充特定材料。如果所有章节完整或已达到最大迭代次数，则输出完整性评估报告并进入 Phase 4。无法找到的内容标记为"待补充"供用户确认。

working 子 Agent 会输出阶段性的分析报告到 3.working/analysis-round-N.md，包含章节覆盖状态、缺失内容清单、完整度百分比等关键指标，供用户审查和决策。

配置定义在 agents/config/chip_spec_gen.working.md。

3.4 design_research.workingresult — 结果输出子 Agent

workingresult 子 Agent 是流程的最终产出专家。在 working 阶段迭代完成后，它将经过整理和完整性验证的内容输出为最终文档。输出采用三种形式：自包含的 HTML 报告（NVIDIA 白色主题风格）、可编辑的 .drawio 架构图源文件、以及支持图片引用的辅助 markdown 文件。

HTML 报告使用 html_chip_design_spec skill 生成，遵循 NVIDIA 白色主题规范（#FFFFFF 背景、#76B900 标志性绿色、#F6F8FA 侧边栏）。每组图表以三种形式同时存在：内嵌 SVG 渲染（视觉可见）、mxGraphModel XML（可通过 draw.io URL 参数在浏览器中编辑）、以及独立的 .drawio 文件（在桌面端完整编辑）。架构图通过 drawio_chip_diagram skill 生成，遵循芯片领域的时钟域配色规范、正交布线和标准图例。

所有输出文件放入 4.result/ 目录并附带时间戳（YYYYMMDD-HHMM），确保版本可追溯。辅助 markdown 文件中正确引用图片路径，支持在 Obsidian 等 md 编辑器中直接预览。

配置定义在 agents/config/design_research.workingresult.md。

4. 工作流程

Chip Spec Gen Agent 的工作流程分为 4 个阶段，每个阶段由对应的子 Agent 执行。整个流程呈流水线结构，数据依次通过 1.planning/ → 2.slice/ → 3.working/ → 4.result/ 目录传递。在 Phase 3（working）和 Phase 2（slice）之间存在可选的迭代反馈回路。

4.1 Phase 1 — 规划（Planning）

用户提交源文档后，chip_spec_gen 协调器首先创建四阶段目录结构，然后委托 planning 子 Agent 处理。planning 子 Agent 读取用户输入的各种格式的源文件（WORD、PDF、markdown、图片等），将其统一转换为内部 md 格式，输出到 1.planning/ 目录。同时，它生成 planning.yml 配置文件，其中定义 max_iterations（默认 5）、source_files（源文件列表）、output_requirements（输出文档的章节结构和格式需求）。用户可在此时审查并修改规划参数。

4.2 Phase 2 — 切片（Slice）

slice 子 Agent 读取 1.planning/ 中的源文档，按章节进行切片处理。它对 md 格式的源文档进行标题解析，以一级和二级标题为边界，将文档切分为独立的片段。每个片段保存为独立的 md 文件，文件名为 slice-<章节序号>-<章节名>.md。图片和表格在切片过程中保持完整：图片文件被复制到 2.slice/images/ 目录并附带上下文标签，表格保留所有行列和数据。此阶段遵循"只提取不修改"原则，不对内容做任何润色或重组。

4.3 Phase 3 — 整理（Working）— 核心迭代逻辑

working 子 Agent 是流程的决策中心，执行"重组→检查→迭代"的核心循环。首先，它读取 2.slice/ 中的全部切片数据，按照 output_requirements 中定义的章节结构重新排列文字和图片，对已有内容做轻度润色确保术语一致性。然后逐章检查内容完整性：该章节是否有对应的文字描述？是否需要图表？内容是否充分？如果发现缺失内容且当前迭代次数 N < max_iterations，working 子 Agent 会生成详细的补材需求（说明缺失的章节、需要补充的内容类型），并返回 Phase 2 让 slice 子 Agent 继续从源文档中提取相关材料。如果所有章节完整或 N >= max_iterations，则输出完整性评估报告并进入 Phase 4。无法找到的内容标记为"待补充"。

4.4 Phase 4 — 输出（Result）

workingresult 子 Agent 根据 working 阶段整理后的内容，生成最终的芯片设计规格文档。HTML 报告通过 html_chip_design_spec skill 生成，采用 NVIDIA 白色主题风格，包含完整的章节结构、内嵌 SVG 渲染图、可编辑的 mxGraphModel XML 和可点击的目录导航。架构图通过 drawio_chip_diagram skill 生成，遵循芯片领域绘图规范。等待用户确认，确认后保存到 4.result/ 目录并附带时间戳。辅助 markdown 文件也同时生成，方便用户在 Obsidian 等 md 编辑器中查看和使用。

5. 接口定义

子 Agent 之间通过标准化的文件系统目录进行数据交换，不直接通信。这种松耦合的设计使得每个子 Agent 可以独立维护和升级，也方便用户在工作流任意阶段介入审查。

6. 输出规范

Chip Spec Gen Agent 最终输出采用三种文件格式，分别满足不同的使用场景。所有文件统一纳入 4.result/ 目录，以时间戳区分版本。

HTML 报告（主要输出）

自包含单文件 HTML，NVIDIA 白色主题。包含完整章节结构、导航目录、内嵌 SVG 架构图，以及可展开的 mxGraphModel XML（支持在浏览器中通过 draw.io URL 参数编辑）。

html_chip_design_spec skill

Draw.io 源文件

每组架构图对应一个独立的 .drawio 文件，可在 draw.io 桌面应用或 app.diagrams.net 中打开编辑。支持二次修改信号连接、模块布局、标注信息。

drawio_chip_diagram skill

辅助 Markdown 文件

支持生成附加的 md 文档（附录、参考手册、数据手册等），内嵌图片引用路径，可在 Obsidian、Typora 等 md 编辑器中直接预览。
