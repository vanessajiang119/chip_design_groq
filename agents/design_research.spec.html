1. 系统概览

Design Research Agent（设计研究智能体）是一个面向 SoC / ASIC / FPGA 芯片设计领域的专业研究自动化系统。该 Agent 的核心使命是：接收用户提出的芯片设计相关研究课题，通过 Planning（规划）→ Research（搜索）→ Working（分析）→ WorkingResult（输出） 四个阶段的多轮迭代，自动完成从需求理解到最终结构化报告输出的完整研究闭环。

系统采用 Orchestrator + Sub-Agent 架构模式，由一个主协调器（design_research）和四个专用子 Agent 组成。协调器负责任务分发、状态管理和迭代控制；四个子 Agent 分别对应研究流程的四个阶段，形成完整的数据处理流水线。各子 Agent 之间通过标准化的文件系统目录进行数据交换，降低耦合度，支持用户在工作流任意阶段介入审查和调整方向。

系统以 1.planning/planning.yml 作为迭代控制中枢，默认支持最多 5 轮 研究迭代（用户可配置）。每一轮研究中，research 子 Agent 根据规划阶段的搜索策略使用 WebSearch、WebFetch、MCP 工具等到多个数据源（IEEE Xplore、GitHub、Web Search、厂商文档等）采集原始材料；working 子 Agent 对数据进行综合分析并决定是否需要进入下一轮迭代。当研究目标达成或达到最大迭代次数后，workingresult 子 Agent 调用 html_chip_design_spec 和 drawio_chip_diagram 技能生成包含架构图的专业 HTML 研究报告和配套的 .drawio 源文件。

该系统方案特别适用于芯片设计领域的方案调研、技术选型分析、竞品研究、标准规范梳理等需要跨来源、多角度综合分析的场景，能够显著减少工程师在信息搜集和整理阶段的手动工作量。系统的多轮迭代收敛机制模仿了人类研究人员的"搜索→阅读→思考→再搜索"的自然工作模式，确保研究结论的深度和完整性。

2. 架构设计

Design Research Agent 采用 Orchestrator（协调器）+ Sub-Agent（子代理） 的架构模式。主协调器 delegator 负责任务分发、生命周期管理和迭代控制逻辑，四个专用子 Agent 各自独立执行一个研发环节。子 Agent 之间通过约定的文件系统目录进行数据交换，不直接通信，确保系统的可扩展性和可维护性。

架构的核心理念是单一职责：每个子 Agent 只关注自己阶段的核心任务，降低耦合度。迭代控制逻辑由 orchestrator 根据 working 子 Agent 输出的决策信号和 planning.yml 中的 max_iterations 参数共同决定。各子 Agent 的配置定义在 agents/config/ 目录下，支持独立维护和版本管理。

Orchestrator 主协调器

负责流程编排与状态管理

  • 接收用户输入并创建任务目录结构
  • 按阶段顺序调度四个子 Agent
  • 解析 working 输出的迭代决策信号
  • 控制迭代轮次（上限由 planning.yml 定义）
  • 管理各阶段之间的数据传递
数据流设计

子 Agent 间通过文件系统交换数据

  • planning → research: 搜索地址列表 + 关键词
  • research → working: 原始材料的 markdown 转换结果
  • working → research: (迭代时) 新增搜索方向和关键词
  • working → workingresult: 最终分析结论和结构要求
  • workingresult → output: HTML 报告 + .drawio 图文件

3. 子 Agent 定义

四个子 Agent 各自承担独立的研究阶段任务，通过标准化的输入输出接口协同工作。每个子 Agent 由 agents/config/ 目录下的独立 .md 文件完整定义，包含 name、description、职责说明和详细规则。

子 Agent职责输入输出配置文件
planning 需求分析 + 搜索策略制定 用户输入 Topic / 文件 planning.yml + 首轮 URL 列表 config/design_research.planning.md
research 多源数据采集与 MD 转换 搜索地址 + 关键词 timestamped .md 文件 config/design_research.research.md
working 数据分析 + 迭代决策 research 产出的 MD 文件 分析报告 + 迭代/结束信号 config/design_research.working.md
workingresult 最终 HTML 报告 + 图生成 working 的分析结论 HTML + .drawio + 辅助 MD config/design_research.workingresult.md

3.1 planning — 规划子 Agent

planning 子 Agent 是研究流程的规划专家。其核心职责是解析用户输入的研究主题和参考文件，生成研究方案和搜索策略。输出文件包括 planning.yml（迭代控制配置，默认最大 5 轮）和 search_strategy.md（搜索策略说明），同时将输入文件的摘要写入 1.planning/ 目录。planning 子 Agent 需要确保搜索策略覆盖多个数据源（IEEE、GitHub、Web、厂商文档等），并为第一轮搜索提供具体可执行的 URL 列表。

# planning.yml — 迭代控制配置 topic: "3D-IC DFT 架构研究" max_iterations: 5 search_sources: - ieee_xplore - github - web_search - vendor_docs first_round_urls: - https://ieeexplore.ieee.org/search/3dic_dft - https://github.com/topics/3dic-test

3.2 research — 研究子 Agent

research 子 Agent 是研究流程的数据采集专家。根据 planning 阶段提供的 URL 列表和关键词，到多个来源采集原始数据。支持的工具包括 WebSearch、WebFetch、MCP 工具等。采集结果需转换为 markdown 格式保存到 2.research/ 目录，文件名需带时间戳和主题标识（如 round-1-architecture.md）。必须保留原始内容的表格结构、图片引用、代码示例和来源 URL。

3.3 working — 分析子 Agent

working 子 Agent 是研究流程的分析与决策专家。读取 2.research/ 目录中的采集数据，提取关键发现，生成带时间戳的阶段分析报告写入 3.working/ 目录。核心职责包括：总结本轮研究发现、识别知识缺口、判断是否需要进入下一轮迭代。如果需要继续迭代，应识别新的研究方向并给出新增 URL 和关键词；如果研究完成，则确认目标已达成并进入 Phase 4。需要检查当前轮次是否已达到 planning.yml 中的 max_iterations 上限。

3.4 workingresult — 最终结果子 Agent

workingresult 子 Agent 是研究流程的最终产出专家。在所有迭代完成后，调用 html_chip_design_spec 技能生成自包含的 HTML 报告文件（NVIDIA 白色主题风格），调用 drawio_chip_diagram 技能生成专业芯片架构图。每组图表必须三种形式同时存在：SVG 渲染（HTML 内嵌） + mxGraphModel XML（HTML 内嵌可编辑） + .drawio 文件（独立可编辑）。同时支持生成附加 markdown 辅助文件。所有文件统一命名版本号，放入 4.result/ 目录。

4. 工作流程

Design Research Agent 的核心工作流程采用 多轮迭代收敛 模型。经过一轮规划后，research 和 working 两个阶段组成一个反复执行的内循环，每次循环都基于上一轮的发现提出新的搜索方向，直到研究结论收敛或达到最大迭代次数。这种设计模仿了人类研究人员的"搜索→阅读→思考→再搜索"的自然工作模式，确保研究结论的深度和完整性。

具体执行流程如下：阶段一（Planning），Agent 解析用户需求，生成 planning.yml 配置文件并输出首轮推荐的搜索地址。阶段二（Research Round N），Agent 按规划地址进行多源数据采集，将原始内容转换为带时间戳的 markdown 文件存入 2.research/。阶段三（Working Round N），Agent 分析 research 产出的材料，提取关键发现并决定是进入下一轮迭代还是结束研究。阶段四（Working Result），Agent 调用 html_chip_design_spec 和 drawio_chip_diagram 技能生成最终的结构化 HTML 研究报告和配套的架构图。

5. 接口定义

各子 Agent 之间的接口通过文件系统目录和标准化文件名约定实现。目录结构强制执行规划→研究→分析→结果的四阶段流程，确保数据流转清晰可追踪。

接口源 → 目标传递内容文件格式
IF-01Orchestrator → planning研究 Topic + 参考文件用户输入
IF-02planning → research搜索 URL 列表、关键词、搜索策略planning.yml + .md
IF-03research → working原始材料的 markdown 转换结果round-N-*.md
IF-04 (迭代)working → research新增搜索方向 + 关键词 + URLanalysis-round-N.md
IF-05 (结束)working → workingresult综合结论 + HTML 结构要求analysis-conclusion.md
IF-06workingresult → 输出HTML 报告 + .drawio + 辅助 MD.html + .drawio + .md

目录结构规范

# 任务目录结构 task_dir/ ├── 1.planning/ │ ├── planning.yml # 迭代控制配置 │ └── search_strategy.md # 搜索策略 ├── 2.research/ │ ├── round-1-architecture.md │ ├── round-2-memory-programming.md │ └── round-3-benchmarks.md ├── 3.working/ │ ├── analysis-round-1.md │ ├── analysis-round-2.md │ └── analysis-conclusion.md └── 4.result/ ├── Report_Name_v1.html # 最终 HTML 报告 ├── Report_Name_v1.drawio # Draw.io 源文件 └── supporting-notes.md # 辅助文档

6. 输出规范

最终输出必须符合以下规范：HTML 报告遵循 NVIDIA 白色主题 风格（#FFFFFF 底色、#76B900 绿色强调、#F6F8FA 侧边栏），使用 html_chip_design_spec 技能生成。每组架构图必须三种形式同时存在：SVG 渲染图（HTML 内嵌可见）+ mxGraphModel XML（HTML 内嵌可编辑）+ .drawio 文件（可独立用 draw.io 编辑）。内容使用中英双语格式——中文段落描述（每段 200+ 字）搭配英文专业术语和缩写。

workingresult 子 Agent 在生成最终报告时，需汇总所有轮次的研究发现，形成完整的对比分析表、架构说明图和技术结论。报告应包含报告概览、架构对比、存储层次、编程模型、性能基准、生态分析、总结与趋势等核心章节。所有文件命名统一版本号（_v1），并存放在 4.result/ 目录下。
