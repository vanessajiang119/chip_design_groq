# 设计模板使用指南 — Design Template Usage Guide

> **版本:** V1.0
> **更新日期:** 2026-05-20

---

## 1. 概述

本目录包含四份设计规格模板文件，构成一套层次化的芯片设计规格体系。模板按照芯片设计的自然分解层次组织，从市场需求到实现细节逐层细化：

| 模板文件 | 文档类型 | 目标读者 | 简介 |
|---------|---------|---------|------|
| `01_product.PRD.md` | PRD — 产品需求规格书 | 产品经理、架构师、项目经理 | 定义芯片产品的市场定位、工艺封装、PPA 目标及项目里程碑 |
| `02_soc_arch.HLD.md` | HLD — SoC 顶层架构设计 | 架构师、设计主管、验证主管 | 描述 SoC 系统级视图 — 子系统划分、互联、存储映射、时钟/复位/电源架构 |
| `03_block_arch.HLD.md` | HLD — 模块架构设计 | 设计工程师、验证工程师 | 描述各 IP/模块的外部接口、总线协议、数据流、配置寄存器 |
| `04_block_micro.LLD.md` | LLD — 模块微架构设计 (AI-Executable) | RTL 设计工程师 | 14 章完整模板：接口时序、FSM 编码、CSR 位域、流水线行为、SDC 约束、DFT 要求 — 可直接驱动 RTL 生成 |

编号 `01`–`04` 反映了自然的阅读与填写顺序：从面向市场的产品定义，逐层细化到实现级的微架构设计。

## 4. 各模板快速参考

### 4.1 `01_product.PRD.md` — 产品需求规格书 (PRD)

| 方面 | 说明 |
|------|------|
| **填写时机** | 项目启动阶段，任何设计工作开始之前 |
| **填写人员** | 产品经理 + 系统架构师 |
| **评审人员** | 架构团队、验证主管、项目经理 |
| **核心章节** | 产品定位、工艺/封装、PPA 目标、I/O 规划、里程碑 |
| **输出 Gate** | Gate 0: Product Definition Complete |
| **预估工作量** | 新芯片 1–2 周，衍生版 1–2 天 |

### 4.2 `02_soc_arch.HLD.md` — SoC 顶层架构设计 (HLD)

| 方面 | 说明 |
|------|------|
| **填写时机** | 产品规格定稿后、模块设计开始前 |
| **填写人员** | 芯片/SoC 架构师 |
| **评审人员** | 产品团队、设计主管、验证主管、项目经理 |
| **核心章节** | SoC 模块框图、子系统概要、存储映射、互联、电源架构、验证特性 |
| **输出 Gate** | Gate 1: Architecture Freeze |
| **预估工作量** | 2–4 周（视复杂度而定） |

### 4.3 `03_block_arch.HLD.md` — 模块架构设计 (HLD)

| 方面 | 说明 |
|------|------|
| **填写时机** | SoC 架构定义完成后 |
| **填写人员** | 模块/IP 设计主管 |
| **评审人员** | SoC 架构师、验证工程师 |
| **核心章节** | I/O 列表、总线接口、模块框图、数据/控制流、CSR 汇总、验证特性映射 |
| **输出 Gate** | Gate 1: Architecture Freeze（逐模块） |
| **预估工作量** | 每模块 1–2 周 |

### 4.4 `04_block_micro.LLD.md` — 模块微架构规格书 (AI-Executable LLD)

| 方面 | 说明 |
|------|------|
| **填写时机** | 模块架构批准后、RTL 编码开始前 |
| **填写人员** | RTL 设计工程师 |
| **评审人员** | 设计主管、验证工程师 |
| **核心章节** | 14 章（详见下文），核心为：接口时序(Ch2)、FSM 编码(Ch4)、流水线行为(Ch5)、CSR 位域(Ch7)、SDC 约束(Ch9) |
| **输出 Gate** | Gate 2: RTL Freeze |
| **预估工作量** | 每模块 1–3 周 |

> **AI-Executable 特性**: 本模板包含 14 个详尽章节，每个章节包含 cycle-level 时序图、bit-level CSR 定义、FSM 状态编码表、完成 SDC 约束模板等内容。AI 阅读填充后的模板可直接生成综合级 Verilog RTL 和完整 SDC 时序约束。详细内容请参考模板本身。|

## 5. 填写顺序

推荐的填写顺序遵循层次化逐级细化原则：

```
01_product.PRD.md  ──►  02_soc_arch.HLD.md  ──►  03_block_arch.HLD.md  ──►  04_block_micro.LLD.md
     (1)                    (2)                   (3)                  (4)
```

**分步说明**：

1. **产品 PRD**（`01_product.PRD.md`）：定义芯片级目标、市场定位、工艺节点、PPA 指标。这是"我们在做什么"的唯一事实来源。

2. **SoC 架构 HLD**（`02_soc_arch.HLD.md`）：设计系统架构——子系统、互联、存储映射、时钟/复位/电源。将产品规格分解为系统架构。

3. **模块架构 HLD**（`03_block_arch.HLD.md`）：对 SoC 架构中定义的每个 IP/模块，定义其外部接口、模块框图、数据流和配置寄存器。每个模块创建一个文件。

4. **模块微架构 LLD**（`04_block_micro.LLD.md`）：对每个模块，细化内部实现——流水线阶段、子模块 RTL 划分、时序、PPA 预算。

> **注意**：步骤 3 和 4 可以在不同模块间并行推进，但每个模块的 04 必须在其 03 完成后才能开始。

## 6. 技巧与常见错误

### 6.1 推荐做法

- **保持模板不变**：将模板文件作为只读参考使用。填写前复制到项目目录。
  ```bash
  cp agents/template/01_product.PRD.md my_project/spec/product_spec.md
  ```

- **更新交叉引用**：如果某章节引用了其他模板文件，在自定义模板名称后更新对应的文件名。

- **统一填写占位符**：HTML 注释（`<!-- ... -->`）标记未填写的字段。评审前通过搜索 `<!--` 可快速找到所有待填项。

- **对模板进行版本管理**：每个模板都有修订历史章节。保持更新以确保可追溯性。

### 6.2 常见错误

| 错误 | 后果 | 预防措施 |
|------|------|---------|
| 跳过产品规格书 | PPA 目标对齐偏差、范围蔓延 | 始终从 `01_product.PRD.md` 开始 |
| 先写微架构再写模块架构 | 接口不一致、返工 | 先完成 `03_block_arch.HLD.md` |
| 未更新交叉引用 | 文档版本混乱 | 使用相对链接，重命名后检查引用 |
| 最终文档中遗留 `<!-- -->` 注释 | 评审时产生歧义 | 提交评审前执行 `grep '<!--' *.md` |
| 整个 SoC 只写一份模块架构文档 | 缺少模块级细节 | 每个模块创建独立的 `03_block_arch.HLD.md` |
| 把模板当作检查清单填写 | 设计决策流于表面 | 每个章节都应包含有意义的分析 |

### 6.3 模板定制

提供的模板已较为全面，团队可根据需要：

- 添加项目专属章节（如车规级芯片的安全手册引用）
- 删除不相关的章节（如非安全芯片的安全章节）
- 添加附录以容纳项目特定数据

> 任何定制都应保持层次化编号体系，以维护交叉引用的一致性。

## 7. 与 chip-pipeline 的集成

四个模板与 chip-pipeline 流水线阶段的映射关系如下：

| 流水线阶段 | 主要模板输入 | 生成输出 |
|-----------|------------|---------|
| `s1_spec_analysis` | `01_product.PRD.md` | 结构化规格书、参数列表 |
| `s2_architecture` | `02_soc_arch.HLD.md` + `03_block_arch.HLD.md`（每模块一份） | 模块框图、接口定义 |
| `s3_rtl_design` | `04_block_micro.LLD.md`（14 章节 AI-Executable 模板，每模块一份） | RTL 代码、测试平台、SDC |

流水线操作详情请参考 [chip-pipeline 用户指南](../userguide.md)。
