# 设计模板使用指南 — Design Template Usage Guide

> **版本:** V3.1
> **更新日期:** 2026-05-22

---

## 1. 概述

本目录包含十份设计规格、验证与跟踪模板文件，构成一套层次化的芯片设计规格与验证体系。模板按照芯片设计的自然生命周期组织，从市场研判到验证签核逐层细化：

| 模板文件 | 文档类型 | 目标读者 | 简介 |
|---------|---------|---------|------|
| `00_market.MRD.md` | MRD — 市场需求文档 | 产品经理、管理层、投资方 | 定义市场机会、商业价值、盈利模式、竞品对标、立项可行性研判 |
| `01_product.PRD.md` | PRD — 产品需求规格书 | 产品经理、架构师、项目经理 | 定义芯片产品的市场定位、工艺封装、PPA 目标及项目里程碑 |
| `02_soc_arch.HLD.md` | HLD — SoC 顶层架构设计 | 架构师、设计主管、验证主管 | 描述 SoC 系统级视图 — 子系统划分、互联、存储映射、时钟/复位/电源架构 |
| `03_block_arch.HLD.md` | HLD — 模块架构设计 | 设计工程师、验证工程师 | 描述各 IP/模块的外部接口、总线协议、数据流、配置寄存器 |
| `04_block_micro.LLD.md` | LLD — 模块微架构设计 (AI-Executable) | RTL 设计工程师 | 14 章完整模板：接口时序、FSM 编码、CSR 位域、流水线行为、SDC 约束、DFT 要求 — 可直接驱动 RTL 生成 |
| `05_project_RTM.md` | RTM — 项目需求跟踪矩阵 | 项目经理、验证主管、质量 | 端到端需求追溯：从 MRD→PRD→HLD→LLD→DV Plan→DV Report 的双向覆盖矩阵 |
| `06_soc_dv_plan.md` | DV Plan — SoC 验证规划 | 验证主管、验证工程师 | 定义 SoC 级验证策略：方法学矩阵、环境架构、场景分解、覆盖率目标、验证里程碑 |
| `07_block_dv_plan.md` | DV Plan — 模块验证规划 | 验证工程师 | 定义模块级验证策略：UVM 环境架构、测试用例分类、覆盖率计划、形式化验证计划 |
| `08_soc_dv_report.md` | DV Report — SoC 验证报告 | 验证主管、项目经理 | SoC 验证结果汇总：覆盖率分析、Bug 统计、性能验证、Sign-off 检查清单 |
| `09_block_dv_report.md` | DV Report — 模块验证报告 | 验证工程师、设计工程师 | 模块验证结果汇总：用例执行、覆盖率闭合、Bug 收敛、质量评估与签核 |

编号 `00`–`09` 反映了自然的阅读与填写顺序：从市场研判出发，经设计规格到验证规划，最终以验证报告收尾。其中 `05_project_RTM.md` 为**贯穿全流程**的跟踪文档，与各阶段并行维护。

## 5. 填写顺序

推荐的填写顺序遵循层次化逐级细化原则：

```
00_market.MRD.md  ──►  01_product.PRD.md  ──►  02_soc_arch.HLD.md  ──►  03_block_arch.HLD.md  ──►  04_block_micro.LLD.md
     (0)                    (1)                    (2)                   (3)                  (4)
                                                                                                  │
                                                                              ┌───────────────────┤
                                                                              ▼                   ▼
                                                                       06_soc_dv_plan.md   07_block_dv_plan.md
                                                                           (5)                  (6)
                                                                              │                   │
                                                                              ▼                   ▼
                                                                       08_soc_dv_report.md  09_block_dv_report.md
                                                                           (7)                  (8)
```

**分步说明**：

0. **市场 MRD**（`00_market.MRD.md`）：开展市场调研和商业论证，输出行业分析、财务测算、可行性研判。这是"为什么做"的唯一事实来源，MRD 定稿后启动 PRD 编写。

1. **产品 PRD**（`01_product.PRD.md`）：基于 MRD 的市场定位，细化工艺、封装、PPA、I/O、里程碑等产品级规格。这是"做什么"的定义文档。

2. **SoC 架构 HLD**（`02_soc_arch.HLD.md`）：基于 PRD 的产品规格设计系统架构——子系统、互联、存储映射、时钟/复位/电源。将产品规格分解为系统架构。

3. **模块架构 HLD**（`03_block_arch.HLD.md`）：对 SoC 架构中定义的每个 IP/模块，定义其外部接口、模块框图、数据流和配置寄存器。每个模块创建一个文件。

4. **模块微架构 LLD**（`04_block_micro.LLD.md`）：对每个模块，细化内部实现——流水线阶段、子模块 RTL 划分、时序、PPA 预算。

5. **SoC 验证规划**（`06_soc_dv_plan.md`）：基于 PRD 和 SoC 架构，定义 SoC 级验证策略——方法学矩阵、验证环境架构、芯片级场景分解、覆盖率目标、回归策略和 6 阶段验证里程碑。

6. **模块验证规划**（`07_block_dv_plan.md`）：基于模块架构和微架构，定义各模块验证策略——UVM 环境、VIP 组件、测试用例分类（CSR/FSM/Datapath/Error）、覆盖率计划和形式化验证计划。每个模块创建一个文件。

7. **SoC 验证报告**（`08_soc_dv_report.md`）：汇总 SoC 级验证执行结果——覆盖率闭合数据、Bug 收敛曲线、性能/功耗验证结果，输出 14 项 Sign-off 检查清单。

8. **模块验证报告**（`09_block_dv_report.md`）：汇总各模块验证执行结果——Feature 完成矩阵、代码/功能覆盖率、Bug 分布与收敛趋势，输出质量评估与签核结论。

> **注意**：步骤 3 和 4 可以在不同模块间并行推进，但每个模块的 04 必须在其 03 完成后才能开始。

> **流程要点**：步骤 0 (MRD) 必须先于步骤 1 (PRD) 完成，因为 01_PRD 的市场定位和 PPA 目标直接来源于 00_MRD 的可行性研判结论。步骤 5 (SoC DV Plan) 必须在步骤 1 (PRD) 和步骤 2 (SoC HLD) 完成后启动；步骤 6 (Block DV Plan) 必须在步骤 3 和 4 完成后启动。步骤 7 和 8 的验证报告则分别在步骤 5 和 6 的执行完成后才开始编写。

> **贯穿步骤**：`05_project_RTM.md` (需求跟踪矩阵) 在步骤 0 (MRD) 完成后创建初始条目，随步骤 1~8 的推进持续更新各列引用和状态，在最终 Gate 5 (Sign-off) 时冻结。RTM 的 §4 (覆盖率统计分析) 是项目健康度的核心仪表盘。

## 6. 技巧与常见错误

### 6.1 推荐做法

- **保持模板不变**：将模板文件作为只读参考使用。填写前复制到项目目录。
  ```bash
  cp agents/template/01_product.PRD.md my_project/spec/product_spec.md
  ```

- **更新交叉引用**：如果某章节引用了其他模板文件，在自定义模板名称后更新对应的文件名。

- **统一填写占位符**：HTML 注释（`<!-- ... -->`）标记未填写的字段。评审前通过搜索 `<!--` 可快速找到所有待填项。

- **对模板进行版本管理**：每个模板都有修订历史章节。保持更新以确保可追溯性。

### 6.2 常见错误

| 错误 | 后果 | 预防措施 |
|------|------|---------|
| 跳过市场需求文档 | 市场定位模糊、商业论证不充分 | 始终从 `00_market.MRD.md` 开始 |
| 跳过产品规格书 | PPA 目标对齐偏差、范围蔓延 | 始终从 `01_product.PRD.md` 开始 |
| 先写微架构再写模块架构 | 接口不一致、返工 | 先完成 `03_block_arch.HLD.md` |
| 未更新交叉引用 | 文档版本混乱 | 使用相对链接，重命名后检查引用 |
| 最终文档中遗留 `<!-- -->` 注释 | 评审时产生歧义 | 提交评审前执行 `grep '<!--' *.md` |
| 整个 SoC 只写一份模块架构文档 | 缺少模块级细节 | 每个模块创建独立的 `03_block_arch.HLD.md` |
| 把模板当作检查清单填写 | 设计决策流于表面 | 每个章节都应包含有意义的分析 |
| 跳过验证规划直接执行验证 | 验证无目标、覆盖率无法闭合 | 在验证执行前完成 `06_soc_dv_plan.md` / `07_block_dv_plan.md` |
| SoC 验证与模块验证范围重叠/遗漏 | 验证效率低或遗漏关键场景 | 明确 06（SoC 芯片级场景）与 07（模块内部逻辑）的分工边界 |
| 设计规格变更后未同步更新验证计划 | 验证与设计不一致，遗漏新功能验证 | 设计变更触发 CR（Change Request）时同步更新 06/07 |
| 验证报告中的 Bug 分类标准不一致 | 无法准确评估设计质量 | 在 06/07 中预先定义 Bug 优先级和分类标准 |
| 覆盖率数据仅看代码覆盖率 | 功能遗漏无法被发现 | 同时追踪代码覆盖率、功能覆盖率（Covergroup）和断言覆盖率 |
| RTM 未随项目阶段及时更新 | Sign-off 时无法提供完整追溯链，审计不通过 | 将 RTM 维护纳入各阶段的 DOR (Definition of Ready) 检查项 |
| RTM 只记录正向追溯 (MRD→DV) | 无法回答"这个验证用例测试的是哪个需求" | 同时维护正向和逆向追溯路径 |
| 使用电子表格等离线工具维护 RTM | 与设计文档脱节，难以同步更新 | 将 RTM 纳入版本管理 (git)，与设计文档保持同步 |

### 6.3 模板定制

提供的模板已较为全面，团队可根据需要：

- 添加项目专属章节（如车规级芯片的安全手册引用）
- 删除不相关的章节（如非安全芯片的安全章节）
- 添加附录以容纳项目特定数据

> 任何定制都应保持层次化编号体系，以维护交叉引用的一致性。
